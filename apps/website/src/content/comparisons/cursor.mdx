## IDE copilot vs. orchestration layer

Cursor and Stoneforge occupy different parts of the AI development workflow. Cursor is where you write and edit code with AI assistance. Stoneforge coordinates multiple agents working on separate tasks in the background. Most teams would use both, not choose between them.

### Cursor: AI-powered IDE

Cursor is a VS Code fork with deep AI integration. It provides real-time code completion, inline chat, multi-file editing via Composer, and an agent mode that can make changes across your codebase. Since version 2.0, it also supports running up to 8 parallel agents in-IDE and cloud-based background agents that clone your repo, work in a VM, and open a PR when done.

Cursor is a polished, mature product with a large user base. Its interactive editing experience is strong, and the learning curve is minimal if you already use VS Code.

### Stoneforge: task orchestration for agents

Stoneforge is a different kind of tool. It doesn't provide an editor or code completion. Instead, it manages a queue of tasks, dispatches them to AI coding agents (Claude Code, Codex, OpenCode, or others), and coordinates the results. Each agent works in an isolated git worktree. Steward agents handle merge and review automatically.

Stoneforge is newer and rougher around the edges. Its strength is structured orchestration: dependency-aware dispatch, role-based agents (Director, Worker, Steward, Daemon), and automatic merge workflows.

### Using them together

The two tools complement each other:

- **Cursor** for interactive work: exploring code, writing features, debugging, prototyping.
- **Stoneforge** for background work: dispatching a batch of tasks to agents while you focus on something else.

```bash
# While you work in Cursor on the main feature...
# Stoneforge handles supporting tasks in the background
sf task create --title "Add unit tests for auth module"
sf task create --title "Update API documentation"
sf task create --title "Refactor database queries for performance"
```

### Where Stoneforge adds something different

Cursor's parallel agents work within the IDE on prompts you provide. Stoneforge manages an entire task lifecycle outside of any IDE: plans with multiple tasks, dependency ordering, automatic agent dispatch, and Steward-managed merge review. If you're coordinating a sprint's worth of work across many agents, that structure helps.

If you're working interactively on one thing at a time, Cursor's built-in agent mode is likely all you need.
