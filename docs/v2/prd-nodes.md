## Settled Canonical Contracts

- Stoneforge V2 is a control plane for supervised, agent-driven software-engineering execution, not a chat app, project tracker, GitHub replacement, or editor-first product.
- The Workspace is the first-slice operational boundary and maps to one GitHub repository.
- Tasks are planning units. Assignments, Sessions, MergeRequests, and Verification Runs are execution records.
- Automations create workflow intent. The Scheduler owns readiness evaluation, queueing, leasing, placement, retries, resume, escalation, and cancellation propagation.
- Host, Runtime, Agent, RoleDefinition, Automation, and Scheduler remain separate concerns.
- GitHub is the first-slice source-control provider; `MergeRequest` is the internal noun and `PR` is the GitHub-facing noun.
- First-slice merge topology supports direct task PRs and optional plan-branch aggregation.
- Policy, authorization, and audit are separate responsibilities. Stoneforge owns policy decisions and AuditEvents.
- Task Progress Records live on Tasks and store Checkpoints and Repair Context. Documents remain reusable workspace context, not hidden task memory.
- First-slice Checkpoints should be bounded restart context, not lengthy resumability documents. A Checkpoint updates the Task's todo or Acceptance Criteria status and may include a short note, key files, and key insights needed to restart efficiently after crash or context exhaustion.
- Checkpoint updates should be submitted through the broader Stoneforge agent CLI/tool surface, not a checkpoint-specific tool. The action validates input immediately and returns actionable errors to the agent.
- Implementing agents may add small task-local todo items through checkpoint updates when they discover necessary implementation substeps, but they may not add or change Acceptance Criteria. Acceptance Criteria are set on Task creation and may only change through Director or human task-edit/review flows.
- Task Progress Summaries should be generated on demand from Task Progress Records for agent prompt assembly and UI overview; they should cite source Checkpoints and Repair Context and should not be persisted as MVP source of truth.
- Same-Task resumed Assignments and repair Assignments should automatically include a generated Task Progress Summary in prompt assembly, with citations to underlying Checkpoints and Repair Context and the ability to inspect more detail.
- Repair Triggers create a new task-owned repair Assignment and Session on the same Task. Repair context and original task context stay colocated at the Task level through Task Progress Records, Task Progress Summary, branch lineage, and MergeRequest history; repair does not continue inside the original Assignment.
- The original Assignment that records or encounters a Repair Trigger should end as `repair_triggered`, not `succeeded`, `canceled`, or `escalated`.
- Entering `repair_required` records the Repair Trigger and attaches the best available Repair Context immediately, while the failure/review/branch-health information is freshest.
- `repair_required` is itself the task state from which the Scheduler dispatches repair work. It should not transition through `ready` just to dispatch repair. `ready` is reserved for non-repair dispatch when no unresolved Repair Context exists.
- A repair Assignment can target one or more unresolved Repair Context items. It may mark each targeted item `resolved`, `unresolved`, `superseded`, or `unsafe`; one repair can fix several related blockers, and later verification/review may make older repair context obsolete.
- Each Repair Context item should have a stable ID and immutable provenance: source trigger type, source object, relevant Assignment, Session, MergeRequest, head SHA, creator or service actor, created time, current status, and status history. Item content should be append-only; resolution, supersession, and unsafe decisions are audited status transitions rather than destructive edits.
- Stoneforge should deduplicate repeated Repair Triggers into the same unresolved Repair Context item when the source and semantic key match, such as the same Provider Check failing on the same head SHA. A new Repair Context item should be created when the source materially changes, including a new head SHA, different failing check, new reviewer Change Request, new branch-health condition, or policy decision with different required action.
- If a new Repair Trigger arrives while a repair Assignment is already running on the same Task, Stoneforge should attach or deduplicate the new Repair Context immediately but avoid starting a second parallel repair Assignment for that Task. The running repair Assignment should continue by default and receive the updated Repair Context on command/context refresh or checkpoint boundary; Stoneforge should interrupt or escalate only when the current repair direction becomes unsafe or an authorized operator explicitly intervenes.
- Repair internals beyond these rules should stay lightweight in the first-slice PRD; the PRD should focus on user-visible recovery behavior, auditability, and scheduler guarantees rather than over-specifying implementation detail.
- Repair Assignments may propose `superseded` or `unsafe`, but Stoneforge validates the transition. `superseded` is valid when newer review, verification, branch-health, or policy facts make the old context obsolete. `unsafe` may come from an agent report and moves the Task toward `human_review_required` unless policy allows another autonomous repair attempt. Authorized humans may also mark Repair Context items superseded or unsafe with audit.
- A repair Assignment reports the status of targeted Repair Context items and whether task work is complete. The Task's next state is based on remaining unresolved or unsafe Repair Context: if none remain and code work is ready for review, `awaiting_review`; if none remain and non-code work is complete, `completed`; if none remain but task work remains, `ready`; if unresolved items remain and autonomous retry is allowed, `repair_required`; if unsafe items remain or retry is exhausted, `human_review_required`.
- Follow-Up Tasks should not automatically inherit the source Task Progress Summary; use explicit Follow-Up Context for terminal-work follow-ups.
- Task Progress Records should store structured bounded Checkpoints and Repair Context only, with references to relevant Assignment, Session, transcript, log, and event records.
- Full provider transcripts should stay in Session or Execution Lineage storage, not be copied into Task Progress Records, because transcripts are high-volume, provider-specific, and may need separate redaction and retention policy.
- Checkpoint and Repair Context redaction is not required in the first slice unless it can reuse the transcript/log redaction path with little added complexity; secrets are not expected to routinely appear in those structured records.
- AuditEvents and durable Workflow Events should be append-only and retained indefinitely by default in local and self-hosted deployments unless an administrator configures retention.
- Workspace policy should include simple retention controls for raw Session transcripts, provider logs, CI logs, Automation payloads, preview logs, and runtime logs in the first slice. Those raw records should have configurable retention with conservative defaults.
- Checkpoints, Repair Context, bounded summaries, lineage, object state, review outcomes, and merge decisions should be retained longer than raw transcripts/logs because they support continuity, compliance, and lineage.
- Stoneforge should apply best-effort automatic redaction for known secret patterns before storing or displaying raw transcripts/logs.
- Automatically redacted transcript/log content should become the canonical stored transcript/log content in the first slice; Stoneforge should not retain an unredacted original by default.
- Authorized users should be able to manually redact transcript/log ranges after storage; the redaction action should emit an AuditEvent with actor, target range, time, outcome, and reason without exposing the sensitive value in the UI path.
- Manual transcript/log redaction should be irreversible in the product flow: the stored/displayed content range is permanently replaced with a redaction marker and metadata.
- Redacted transcript/log ranges should remain visible as placeholders in Session and Execution Lineage views, with authorized metadata such as actor, timestamp, and reason.
- Restoring redacted content should require external backup/restore outside normal Stoneforge UI and API flows.
- Any future forensic raw-retention mode should be explicit, high-risk, self-hosted policy with separate access controls and audit.

## Decisions From Grilling

- Session progress estimate: after more than 100 questions, the PRD discovery is not as complete as the core workflow depth suggests. Core workflow, policy, execution, repair, routing, and GitHub issue intake are heavily covered, but Automations, durable automation workflows, Documents, document search/categorization/agent behaviors, and CI/CD integration still need substantial grilling before PRD synthesis.
- The PRD should cover the entire first slice defined in `docs/v2/README.md`, not only the direct task execution subset.
- The MVP milestone is the entire first slice. Do not split plan aggregation, host/sandbox paths, user-defined automations, or recovery hardening into post-MVP follow-on milestones if they are part of the first slice docs.
- The first slice should support one primary GitHub repository per Workspace. Cross-repository Workspaces are out of scope and should be deferred to a later slice.
- The first slice should include a simple Org container because the domain model uses Org: create/select Org, manage Workspaces, basic membership/roles, and GitHub org/repo permission projection. Advanced org billing, SSO enforcement, audit export, cross-Workspace policy inheritance, and org-wide analytics are out of scope.
- The first slice should require a guided onboarding wizard based on the `reference/smithy-next` prototype onboarding flow. The wizard should create or select the first Org, create the Workspace, connect the GitHub repository, bootstrap default RoleDefinitions, configure the first Runtime path, configure the first Agent, select policy, and surface readiness blockers. Additions to the prototype onboarding flow are allowed for missing first-slice requirements, but existing prototype setup steps should not be removed.
- The primary first-slice golden path should be: connect a GitHub repository; configure one usable Host or managed sandbox Runtime and Agents; create or import engineering intent; have the Director decompose it into a Plan and Tasks; dispatch Worker Agents to implement each code-changing Task on task branches; create a GitHub PR/MergeRequest for each completed code-changing Task; review task PRs through Review Agents and any policy-required Human Reviews; run previews and observe verification; repair when review, verification, mergeability, policy, or branch health requires it; merge task PRs according to the workspace Merge Topology; for plan aggregation, merge task PRs into the plan branch, update the plan PR from draft to active/ready-for-review when all required Tasks are complete, review the plan PR with Reviewer Agents and any required Human Reviews, then merge the plan PR to the workspace target branch with lineage and audit visible.
- First-slice success metrics should be operational completion metrics, not growth metrics: onboard one real repo to `ready`; create or import intent; have the Director create a valid Task or Plan; have a Worker complete a code change; open a PR; evaluate required verification and review gates; handle at least one failure through repair; merge with `stoneforge/policy`, audit/lineage, Documents context, and GitHub sync visible. A core success metric is that an operator can understand why work is blocked.
- First-slice acceptance should include automated product coverage for the golden workflow through end-to-end or integration tests: onboarding a real GitHub-backed Workspace, configuring Runtime/Agent/RoleDefinition/policy, creating or importing intent, Director Task/Plan creation, Worker execution, PR creation, Verification Run observation, Review Outcome, at least one repair loop, and merge/policy evaluation. The PRD should not prescribe exact test tooling beyond repo standards, but acceptance requires a repeatable proving scenario that exercises real GitHub integration rather than fake providers.
- First-slice acceptance should require a lightweight security checklist for the concrete trust surfaces in scope: GitHub App permissions and token lifetime, OAuth identity linking, RBAC/reviewer eligibility, Session Command Credentials, AutomationRun Credentials, webhook signing and idempotency, secret-backed provider and registry proxies, preview/dev secret warnings, policy-bypass paths, AuditEvents for sensitive actions, and raw log/payload retention and redaction. Formal SOC2/compliance certification, penetration testing, and a full threat-model program are out of scope for the first slice.
- Remaining PRD discovery should stop broad grilling and close only high-risk gaps needed for synthesis, especially data model/API boundaries, security/privacy retention, deployment/local install expectations, and explicit out-of-scope decisions.
- First-slice product-level API scope should be required only where external actors depend on contracts: the Agent Command Surface, Stoneforge SDK and Automation handler APIs, GitHub webhook/provider integration boundaries, and minimal UI/backend private APIs needed by the app. Do not require a fully public REST or GraphQL API for every object in the first slice. Internal domain and persistence APIs remain implementation detail, with type-driven contracts and normalized data model expectations documented for engineering.
- A Workspace should be considered ready for the golden path only when it has a connected GitHub repo, target branch and merge topology, sufficient GitHub App permissions for branches/PRs/checks, at least one healthy Runtime path, at least one enabled Agent with an acceptable Runtime, an effective policy preset, and visible readiness status for provider checks, previews, and secrets. Default RoleDefinitions for Director, Worker, and Reviewer workflows should be auto-created by the platform during Workspace bootstrap; users may later modify or delete them.
- First-slice RoleDefinitions should be editable Workspace objects with simple version/history tracking. The platform should create default Director, Worker, and Reviewer RoleDefinitions, and authorized users can modify, disable, delete, or create RoleDefinitions. Assignments should snapshot or reference the RoleDefinition version they started with so later edits do not rewrite historical execution. Advanced marketplace roles, role-template libraries, multi-version rollout controls, role A/B testing, and cross-workspace role sharing are out of scope.
- If a user deletes or breaks a default Director, Worker, or Reviewer RoleDefinition, the Workspace should stay `ready` as long as repo connectivity and at least one execution path remain valid. Stoneforge should instead show workflow-level readiness blockers for affected workflow categories, such as blocking automated review dispatch when no usable Reviewer RoleDefinition exists.
- GitHub Issue import/sync is required in the first slice as an engineering-intent source, but issue sync must not mean unconditional executable Task creation. By default, GitHub Issues should import as intent or issue-backed Draft Tasks and require policy-controlled Director triage before becoming dispatchable.
- GitHub Issue sync should be bidirectional when enabled by policy: linked Stoneforge Tasks may create or update GitHub Issues, and linked GitHub Issues may update Stoneforge intent/task metadata. The relationship is not required to be 1:1: one GitHub Issue may decompose into multiple Tasks, and one Task may have one primary Issue plus optional secondary links.
- Workspace policy should control GitHub Issue intake behavior, including ignore/import-only, Director triage, Draft Task creation, or automatic Task promotion for trusted authors, teams, labels, milestones, or repositories. Public or untrusted Issues should default to import-only or Director triage to avoid external issue creation triggering dispatchable work.
- Minimum first-slice GitHub Issue sync should include title, body, author, labels, assignees, milestone, open/closed state, URL/number, timestamps, and comments sufficient for Director triage context. Stoneforge should write back a linked-task comment or status comment, with optional label/status updates when policy enables them. PRs created from linked Tasks should reference the primary linked Issue and relevant secondary Issues. Task PRs may use issue-closing keywords only when the linked Issue has exactly one linked Task; when an Issue decomposes into multiple Tasks, task PRs use non-closing references and the Plan PR should link back to the source Issue for plan-level closure. GitHub PR review comments and top-level PR comments should be visible enough in Stoneforge to support review and repair context. Full bidirectional field-perfect sync, GitHub Projects sync, issue forms/schema mapping, full bidirectional comment sync, threaded resolution parity, perfect comment editing/deletion sync, and complex conflict resolution are out of scope.
- When policy enables Stoneforge label writeback, Stoneforge may manage reserved labels on ingested Issues such as `stoneforge:status:backlog`, `stoneforge:status:todo`, `stoneforge:status:in-progress`, `stoneforge:status:review`, `stoneforge:status:done`, `stoneforge:priority:{priority}`, and `stoneforge:complexity:{complexity}`. These labels mirror Stoneforge state for GitHub visibility; Stoneforge remains the workflow source of truth. In the first slice, status, priority, and complexity labels are Stoneforge-owned projections, not GitHub-side commands. If edited directly on GitHub, Stoneforge should reconcile them back or surface a quiet conflict rather than treating GitHub labels as a second workflow source of truth.
- First-slice policy presets should stay simple: only `autopilot` and `supervised`.
- `backlog` should not become a separate canonical Task state in the first slice. Backlog is UI/product grouping language for Tasks that are not ready to dispatch, overlapping with `draft` or other non-dispatchable states.
- Issue Intake Policy defaults should depend on policy preset. Autopilot defaults to import/sync plus Director triage. Supervised defaults to import/sync Issues as non-dispatchable draft Tasks, which may be shown in a backlog UI group. Draft/non-dispatchable Tasks cannot become ready without a human status change or a human instruction to send them to Director triage.
- Trusted maintainers may apply reserved labels to override the default intake path: `stoneforge:auto-dispatch` promotes the imported Issue directly to a ready Task, and `stoneforge:auto-triage` sends it directly to Director triage. A separate policy may prevent trusted-maintainer label overrides, and that prevention policy defaults to false for all presets.
- `stoneforge:*` labels are ordinary labels set directly on GitHub Issues, but Stoneforge honors their intake effects only when the actor who applied the label is a linked trusted maintainer. Labels applied by unlinked users, external contributors, or actors without sufficient GitHub-backed repository permission remain visible provider facts but are ignored for Stoneforge intake automation.
- When Stoneforge ignores a `stoneforge:*` label because the actor is untrusted, unlinked, or lacks sufficient permission, it should show a quiet UI warning on the imported Issue or Task intake record and record workflow/audit lineage. It should not post a public GitHub comment by default; Workspaces may optionally enable maintainer-only notifications.
- Externally created GitHub Issues should always default to imported non-dispatchable draft state, shown in backlog/intake UI with a clear external-origin indicator visible in the UI and included in Director triage context, so humans and Directors apply more care before creating or activating work from them.
- A separate Imported Issue Triage Approval policy should define whether imported Issues require human approval before Director triage: no Issues, only externally created Issues, or all Issues. Defaults are no Issues for autopilot and externally created Issues for supervised.
- The first-slice MVP must support local OSS/dev and self-hosted control-plane deployments as first-class paths. The architecture should remain cloud-compatible, but the PRD should not require a fully operated Stoneforge cloud service, SaaS billing, or hosted-tenant operations.
- A seeded local demo/sandbox Workspace may be included for local OSS/dev mode if cheap, but it should not be part of core MVP acceptance criteria. The required path is real GitHub-backed onboarding through the onboarding wizard.
- First-slice local OSS/dev should run with SQLite or local Postgres, a clearly marked single-user principal, local filesystem/dev secret storage, and real GitHub App configuration for repository testing. Fake providers are not useful for the local product contract and should not be part of first-slice acceptance.
- First-slice self-hosted team mode should require Postgres, durable object or file storage for logs and artifacts as needed, a configured GitHub App, OAuth or OIDC human authentication, secret storage, worker and Host connectivity, and clear environment readiness checks. Kubernetes, high availability, backup/restore automation, managed upgrades, billing, multitenant SaaS operations, and a Stoneforge-hosted cloud service are out of scope.
- First-slice release acceptance should prove local/self-hosted operability, not only document it. Require repeatable local OSS/dev setup and repeatable self-hosted team setup with documented environment variables, GitHub App setup, database/storage setup, secret storage expectations, Host/Runtime connectivity checks, health/readiness checks, and troubleshooting for common misconfiguration. One-click installers, Kubernetes manifests, HA topology, backup automation, managed upgrades, and production SRE runbooks are out of scope.
- Explicit first-slice exclusions include multi-repo Workspaces, non-GitHub source-control providers, GitHub Projects sync, full comment/thread resolution parity, provider-specific CI integrations beyond GitHub checks/statuses/logs available through GitHub APIs, native CI hosting, remote GitHub Actions rerun or workflow dispatch controls, semantic/vector Document search, rich binary Document editing, cloud SaaS operations, billing, SSO enforcement, high availability, backup/restore automation, arbitrary public REST or GraphQL APIs for every object, mobile apps, and advanced analytics/reporting. Scope should stay limited to the paths required for the first-slice golden workflow, onboarding, review, repair, merge, Automations, Documents, and local/self-hosted operation.
- First-slice notifications should include only in-app notification and action-required surfaces needed for operators to notice blocked or action-required work: readiness blockers, human review needed, policy-required approval or activation, failed required verification, repair escalation, blocked or failed AutomationRuns, Host or Runtime disconnected, and missing configuration. Email, Slack, webhook notification delivery, digest preferences, escalation schedules, and notification routing rules are out of scope unless already covered by user-defined Automations.
- Existing Stoneforge V1 or Smithy data migration/import is out of scope for the first slice. `reference/v1` and `reference/smithy-next` are reference sources only: V1 for historical behavior/code context and smithy-next for UI/workflow direction. First-slice setup should create clean V2 Orgs, Workspaces, policies, Agents, Runtimes, Documents, Automations, Tasks, and Plans rather than importing legacy production data. A later migration slice can be designed once V2 canonical objects stabilize.
- Broad Workspace object export/import is out of scope for the first slice. Require normal UI access and internal persistence for first-slice objects, plus GitHub-backed artifacts where applicable. Audit export, full Workspace export/import, backup/restore automation, portable project archives, and migration tooling are out of scope. Minimal JSON/debug exports for local development may exist as engineering conveniences, but they are not product acceptance requirements.
- Customer-managed Host execution and Daytona managed sandbox execution both need parity at the Stoneforge lifecycle/model boundary: Runtime, Agent, RoleDefinition, Scheduler, Assignment, Session, checkpoint, outcome, PR, review, and audit behavior must work through both paths. They do not need identical provider-specific operational knobs in MVP.
- Claude Code and OpenAI Codex must both be first-class Worker Agent backends for the full task execution lifecycle. Shared adapter parity must include launch, resume, cancel, provider session identity capture, heartbeats/logs, validated checkpoint update capture, final outcome reporting, sanitized audit/telemetry correlation, RoleDefinition-driven prompt/config injection, tool access, skill/MCP access, and lifecycle hooks. Provider-specific transcript richness and UX affordances may differ if the Stoneforge Assignment/Session contract remains identical.
- Automated golden-workflow acceptance should use both deterministic and real provider execution paths. A deterministic test Agent/adapter may prove stable CI coverage of Stoneforge workflow state machines, GitHub PR flow, policy, review, repair, and merge behavior. Separate first-slice release smoke coverage should prove real Claude Code and OpenAI Codex adapter launch, resume, cancel, Session identity, progress, and outcome paths against real providers or their supported local app-server integration. Do not make every golden-path test depend on live LLM quality, latency, or cost.
- The first-slice MVP must include a real Director Agent workflow. The Director Agent must be autonomously capable and strong at clarifying engineering intent, producing Tasks with acceptance criteria and dependencies, optionally producing a Plan, and moving the graph toward execution. It does not need perfect decomposition quality.
- The first-slice PRD should include explicit agent quality and autonomy non-goals. Stoneforge must orchestrate real Director, Worker, and Reviewer Agents through durable workflows, but it does not promise perfect task decomposition, perfect code generation, perfect review accuracy, zero human intervention, or agents always producing ideal work on the first try. Acceptance should focus on bounded task quality, clear Acceptance Criteria, recoverable failures, visible blockers, review and verification gates, repair loops, and audit/lineage.
- The first-slice PRD should require a simple Workspace policy configuration surface, not a general policy language or editor. It should support choosing `autopilot` or `supervised`, configuring the first-slice policy values already identified, and showing effective policy/readiness consequences. A generic rule builder, scripting policy language, org-wide inheritance UI, policy simulation, policy version diffing, and bulk policy management are out of scope.
- Director-created Plans should not require human activation before planned tasks dispatch as a hard product rule. Plan activation may be autonomous when policy allows; human approval/intervention is policy-controlled rather than always required.
- Activation review policy should stay simple. Workspaces may choose whether task or plan activation requires human review/approval using risk-level modes: never require, require only for high-risk or sensitive tasks, or always require.
- Missing acceptance criteria, ambiguous dependencies, broad scope, and role/capability fit should be resolved by the Director workflow rather than exposed as separate activation policy thresholds.
- The Director may escalate and wait for human approval when confidence is low. The workspace policy should define the Director confidence level that requires human approval.
- Task and Plan risk classification should be an explicit Director-produced field with a small enum: `low`, `medium`, `high`, plus a separate `sensitive` boolean or reason list. The Director must justify `high` or `sensitive` classifications in structured text. First-slice policy should branch only on `high` or `sensitive` versus all other work.
- A Director-produced Plan Graph is coherent when every Task has intent, acceptance criteria, risk classification, required RoleDefinition category or tag constraints, and dependency edges that are acyclic and reference existing tasks or plans; every blocking dependency has a clear completion condition; merge topology is selected; and policy evaluation for activation passes or enters the configured approval path.
- A coherent Plan Graph does not need at least one newly created task that is immediately dispatchable. It may depend entirely on existing Tasks or Plans already on the board, as long as those dependencies are explicit and resolvable.
- First-slice dependencies may connect Tasks and Plans. A Task may depend on a Task or Plan, and a Plan may depend on a Task or Plan for activation/readiness.
- Dispatch and execution remain task-centered. Plans do not directly own implementation execution. A Plan dependency is satisfied only when the dependency Plan reaches its policy-defined completion state, normally `completed`.
- Planned code-changing work should default to plan aggregation: task branch to plan branch PR, then plan branch to workspace target branch PR.
- Direct-to-target planned tasks should remain supported as a toggleable workspace policy.
- Plan-level review feedback should create repair context on the Plan MergeRequest and require the Director Agent to map that feedback to one or more new or existing Tasks inside the Plan.
- Coding repair from plan-level feedback dispatches as Task-owned Assignments. If feedback is purely integration or merge-topology related, the Director may create an integration-repair Task inside the Plan. Plans must not receive generic Worker coding Assignments directly.
- User-defined automations in the first-slice MVP must support curated product-event triggers, schedules, and inbound signed webhooks. The curated trigger list should include Task state changed, Plan state changed, MergeRequest opened or updated, Review Outcome recorded, VerificationRun changed, AutomationRun changed, schedule, and inbound signed webhook.
- Inbound webhook triggers should use one unique endpoint per Automation in the first slice. This is easier to display, rotate, disable, and audit. The endpoint should still require a signed request and idempotency key. A shared workspace endpoint may come later if needed.
- Inbound webhook signing should support both per-webhook signing secrets and a workspace-wide signing secret. Per-webhook secrets provide isolation, while a workspace-wide secret avoids environment-variable sprawl for Workspaces with many inbound webhook Automations. Workspace or Org policy may disable workspace-wide signing-secret use when stricter isolation is required.
- Inbound webhook signing secrets, both per-webhook and workspace-wide, should support rotation with an overlap window where old and new secrets validate. Stoneforge should show last-used metadata so operators can confirm clients have switched before retiring the old secret.
- Outbound automation webhook signing should mirror inbound: support per-destination signing secrets and workspace-wide outbound signing secrets. Workspace or Org policy may require per-destination secrets for stricter isolation. Rotation should support an overlap window and last-used metadata.
- Arbitrary "any object changed" Automation triggers are out of scope for the first slice until event schemas and loop-prevention rules are stronger.
- AutomationRun changed triggers are allowed but require guardrails: explicit trigger filters for AutomationRun state, action, and source; prevent an Automation from triggering directly on its own runs by default; enforce idempotency keys; and cap automation chain depth per root correlation ID.
- When an Automation chain exceeds the chain-depth cap, Stoneforge should create a blocked AutomationRun with reason `automation_chain_depth_exceeded`, record lineage to the root correlation ID and parent run, and surface it in the Automations UI. It must not be silently dropped. Manual re-run may be allowed after inspection or edit, subject to normal policy.
- User-defined automation chain-depth cap should be configurable by Workspace policy, with a conservative default of `3`. Platform Automations may use separate internal safeguards, but user-defined automation chains need a workspace-visible cap.
- Schedule-based Automations should support both simple intervals and cron expressions in the first slice. Timezone should be explicit and required. Each schedule trigger should record evaluated next-run time and timezone in AutomationRun/source metadata.
- Missed schedule runs after downtime should not be backfilled in the first slice. Stoneforge should evaluate the next future run and record a missed-run event/count for visibility. Users can manually run the Automation if they need to recover missed work.
- Manual runs of schedule-based Automations should create separate AutomationRuns and must not move or reset the next scheduled run time.
- User-defined automation actions in the first-slice MVP must support creating scheduler-evaluated Agent/workflow intent with an explicit RoleDefinition and calling outbound signed code-first webhooks with idempotency.
- Stoneforge must not host arbitrary user workflow code inside the first slice.
- The first-slice automation model should have only two action types: agent-based automations and code-first automations.
- `workflow-evaluation` should not be modeled as a third automation type. Built-in merge-evaluation and failure-escalation behavior should be platform-defined code-first automation, even when it runs internal Stoneforge code rather than user-hosted code.
- Mandatory platform-provided automations are Director-intent decomposition dispatch, ready-task dispatch, PR-created review dispatch, change-request repair dispatch, optional-provider-check Director triage dispatch, platform code-first merge evaluation, and platform code-first failure escalation.
- Platform-defined and user-defined code-first automations should share the same conceptual boundary: trigger context, target object, policy context, correlation/idempotency key, and a bounded action result.
- Platform-defined code-first handlers may run in-process and call internal control-plane operations directly. User-hosted handlers receive equivalent context over signed outbound webhooks and use the Stoneforge API for workflow actions they need during execution.
- Neither platform-defined nor user-defined code-first automation may mutate workflow state outside policy-checked and audit-recorded control-plane operations.
- Automations should have first-class durable `AutomationRun` records in the first slice. Every automation trigger evaluation should record trigger source, evaluated policy, idempotency key, target object, action type, status, attempts, timestamps, result, and linked follow-up objects.
- Agent-based AutomationRuns may create Dispatch Intent and later Assignments/Sessions, but the AutomationRun remains the durable explanation of why the workflow started. Code-first AutomationRuns record their own attempts and results without pretending to be agent execution.
- AutomationRun states should use one unified success state rather than action-specific success states: `created`, `blocked`, `delivering`, `retry_wait`, `running`, `succeeded`, `failed`, and `canceled`.
- `succeeded` means the automation action itself succeeded. For agent-based runs, the successful output is usually one or more linked Dispatch Intents; downstream Assignment/Session completion belongs to those linked records. For code-first runs, outputs may include provider facts, workflow state updates, webhook acknowledgements, or Dispatch Intents created as follow-up work.
- `blocked` is not terminal by definition. It means policy, eligibility, idempotency, or missing configuration currently prevents action. A blocked AutomationRun may be continued or re-evaluated when the blocking facts change, according to retry/resume policy.
- `delivering` and `retry_wait` are for outbound delivery attempts. `running` is for long-running code-first workflows that were accepted and are awaiting signed callback completion.
- Outbound automation webhook delivery should retry automatically for retryable responses. Treat `2xx` as delivery success; `408`, `429`, and `5xx` as retryable; and other `4xx` responses as terminal failure. Use exponential backoff with a bounded retry count. For long-running workflows, delivery success moves to `running` only when the response indicates accepted/async; otherwise a bounded synchronous `2xx` can complete the run as `succeeded`.
- Outbound handler responses should use an explicit typed response contract, not status-code inference alone: for example `202` with `{ "status": "accepted" }` transitions to `running`, `200` with `{ "status": "succeeded" }` completes as `succeeded`, and `200` with `{ "status": "failed" }` completes as `failed` with result details. Stoneforge should provide SDK helpers so users can return accepted/succeeded/failed responses without memorizing status codes or payload shapes.
- Stoneforge should provide a broader TypeScript-first Stoneforge SDK, recommended but not required, for interacting with the Stoneforge API. It should include Automation handler support: signature verification helpers, typed response helpers, callback helpers, Stoneforge API client setup using the AutomationRun Credential, and thin helpers for common API actions such as create Task, create Plan, request Director triage, link objects, complete or fail an AutomationRun, and read scoped context. These helpers are wrappers over the documented API, not a separate workflow DSL.
- The outbound automation webhook protocol should be language-agnostic. The first-slice Stoneforge SDK should ship for TypeScript first, with other language SDKs left for later or manual protocol implementation.
- The TypeScript Stoneforge SDK is in first-slice scope, with a tight required surface: auth/client setup, Automation webhook signature verification, typed accepted/succeeded/failed responses, callback helpers, and the common API wrappers identified above. Keep broader SDK expansion out of scope unless needed by first-slice workflows.
- Blocked AutomationRun re-evaluation should be controlled by explicit Workspace policy, not by a separate automation risk/sensitivity taxonomy. Policy modes should be: auto re-evaluate platform and non-external user-defined automations only; auto re-evaluate all blocked automations; or auto re-evaluate none. The default should re-evaluate platform automations and non-external user-defined automations, but not externally-triggered automations.
- Long-running code-first AutomationRuns should complete through signed callbacks only in the first slice. Stoneforge should not poll external status endpoints. Callback receipt must return a 200 response after the callback is authenticated, idempotently accepted, and durably recorded, so external handlers can safely retry missed or failed callbacks.
- Completion callbacks finalize AutomationRuns only. They report `succeeded` or `failed`, bounded result metadata, linked objects, and error details. They are not a general workflow mutation surface; user-defined code-first automation should use the Stoneforge API for workflow actions.
- External code-first automation should call the Stoneforge API with a short-lived scoped AutomationRun Credential minted by Stoneforge and included or retrievable from the outbound webhook context. It should be scoped to the Workspace, AutomationRun, configured target objects, allowed API action categories, and expiry. It is not a human credential or broad service token, and every API call must emit service-actor lineage.
- AutomationRun Credentials should be usable from outbound delivery through the `running` state, including final API calls before completion callback. They are revoked when the AutomationRun reaches any terminal state such as `succeeded`, `failed`, or `canceled`.
- User-defined code-first automation may create Tasks and Plans through the Stoneforge API when its AutomationRun Credential scope allows it. There should not be a hard default requiring those Tasks or Plans to be created as drafts. Whether automation-created Tasks or Plans are accepted and move forward, require Director triage, become draft/non-dispatchable, or become ready/active should be controlled by Workspace policy or the Automation's configuration, with the default accepting created Tasks/Plans and moving them forward.
- Workspace policy is the ceiling for automation-created Task/Plan behavior, and Automation configuration can only narrow it. If Workspace policy allows automation-created work to become ready/active, a specific Automation may still choose Director triage or draft/non-dispatchable. If Workspace policy requires Director triage, the Automation cannot bypass it.
- Requesting Director triage for automation-created Tasks or Plans should be an explicit Stoneforge API action, such as `request_director_triage`, authenticated by the AutomationRun Credential when scope allows. The request creates scheduler-evaluated Dispatch Intent for the Director RoleDefinition unless policy requires human approval first, in which case the request becomes pending or blocked.
- `request_director_triage` may target objects created by the same AutomationRun or existing Tasks, Plans, and imported Issues. Eligibility is controlled by AutomationRun Credential scope over target objects or object classes, not by requiring the automation to have created the object.
- Repeated `request_director_triage` calls should be safe and deduped by target object, triage purpose or reason, source Automation/AutomationRun, and unresolved Director Dispatch Intent status. If matching triage is already queued or active, the API returns the existing linked Dispatch Intent instead of creating duplicate Director work.
- Users should be able to manually re-run an AutomationRun from any terminal state, including `succeeded`, `failed`, `canceled`, or still-blocked runs when appropriate. Manual re-run always creates a new AutomationRun linked to the original; it never mutates the old run.
- Manual re-run should default to replaying the original trigger context exactly. A separate run-with-edited-inputs path may create a new manual AutomationRun with the input diff recorded; editing inputs requires the same authorization as creating or editing the Automation.
- The first-slice Automations UI should include an Automation list/detail view and AutomationRun history. AutomationRun history should be searchable/filterable by Automation, target or linked object, trigger type, action type, state, source actor/provider, time window, and correlation or idempotency key.
- AutomationRun detail should show raw trigger, outbound request, and callback payloads in the first slice. Do not add special first-slice permissions or redaction for these payloads; secrets should not be placed in workflow triggers, outbound automation requests, or callbacks. Automation payloads remain raw records subject to Workspace retention policy.
- Stoneforge should document that automation payloads must not contain secrets and use lightweight warnings for obvious secret-looking field names such as `secret`, `token`, `password`, `apiKey`, or `privateKey`. Do not build a heavy DLP system for automation payloads in the first slice.
- Automation definitions should be versioned. Each AutomationRun should snapshot the effective Automation definition or reference an immutable Automation version at run creation. Later edits must not change historical run interpretation or exact manual replay behavior.
- Editing an Automation definition affects only future AutomationRuns. Existing `running`, `delivering`, `blocked`, or `retry_wait` AutomationRuns keep the version/snapshot they were created with. To use the new version for stuck or old work, an operator cancels or re-runs, creating a new linked AutomationRun.
- Disabling an Automation prevents new trigger-created AutomationRuns but does not stop existing runs by default. Existing runs continue on their pinned version unless an operator explicitly cancels them. The UI should warn when disabling an Automation with active runs and offer an optional cancel-active-runs action.
- Automation hard delete should be out of scope for the first slice. Use archive or soft-delete semantics instead: hide the Automation from active lists, prevent new runs, and preserve Automation versions plus historical AutomationRuns for audit, replay, and debugging.
- Archived or soft-deleted Automations should not allow direct manual re-run of historical AutomationRuns by default. Historical runs remain inspectable, but rerun requires restoring/unarchiving the Automation or explicitly creating a new Automation from the historical version.
- `supervised` is a policy preset, not a separate policy system. Its default policy values should require at least one qualifying Required Human Review for code-changing merge.
- `autopilot` is a policy preset whose default values should require Required Agent Review for code-changing merge, plus verification, mergeability, and policy pass.
- Review and approval are not separate record types in the current model. A passing review is a `Review Approved` outcome, and Required Reviews decide which `Review Approved` outcomes qualify.
- A Human Review Approved outcome does not need to wait for Verification Runs or Agent Review to pass by default. Policy may optionally require verification, agent review, or other conditions before a Human Review Approved outcome may be recorded or before it can satisfy merge readiness.
- Human Review Approved outcomes may be recorded while required Provider Checks are pending by default. They do not satisfy merge readiness until policy-required verification and other gates pass. Policy may optionally require verification before a Human Review Approved outcome may be recorded or counted.
- Policy should distinguish between recording a `Review Approved` outcome and satisfying a Required Review. Recording is usually permissive; required-review satisfaction is stricter and only contributes to `merge_ready` once policy-required conditions for that requirement are met.
- Required Review freshness should follow GitHub-compatible semantics in the first slice. For GitHub-backed Workspaces, GitHub branch protection/rulesets are the source of truth for stale-approval dismissal and most-recent-reviewable-push requirements. Stoneforge may cache and display the observed settings, but should not allow conflicting local overrides for GitHub-backed repos in the first slice. Local and non-GitHub modes may use equivalent Workspace policy settings. When stale-approval dismissal is enabled, a diff-changing push or merge-base change after a Review Approved outcome makes that approval stale for Required Review satisfaction while preserving it in history. When stale-approval dismissal is not enabled, prior approvals may remain eligible across later commits unless a most-recent-reviewable-push rule requires approval after the latest reviewable push by someone other than the pusher.
- GitHub branch protection/ruleset changes while a MergeRequest is open should be treated as merge-readiness input changes. Stoneforge should refresh or observe the provider settings, re-evaluate Required Review satisfaction, and republish `stoneforge/policy` if the decision changes. This should not create repair work because it is a policy/merge-readiness change, not a code defect.
- The first slice should refresh GitHub branch protection/ruleset settings during PR observation and merge evaluation rather than continuously monitoring them. The UI should show the last observed settings and refresh time, with a manual refresh action when needed.
- Stoneforge should import enough GitHub review, reviewer identity, and branch protection/ruleset data to decide Required Review satisfaction. It should not attempt to fully mirror GitHub's reviewer request UI, CODEOWNERS assignment behavior, or team notification workflow beyond displaying useful context.
- Imported GitHub reviews may satisfy Stoneforge Required Reviews only when the GitHub reviewer identity is linked to a Stoneforge human user who is authorized for the relevant requirement. Unlinked GitHub reviews may be displayed or recorded as provider context but do not satisfy required reviews.
- Human Review Outcomes recorded in Stoneforge should be posted back to GitHub as PR reviews only through the Human Reviewer's linked eligible GitHub OAuth identity. Human Review Approved maps to a GitHub approve review and Human Change Request maps to a GitHub request-changes review. Without an eligible linked GitHub OAuth identity, the Stoneforge review may be recorded as context but does not satisfy GitHub-side or Stoneforge Required Human Review for that repo. Stoneforge should not imply that the GitHub App can impersonate a human reviewer.
- GitHub OAuth linking for humans should be just-in-time, not a Workspace onboarding blocker. Workspace onboarding may install the GitHub App and connect repositories without every human linking OAuth. When a human attempts a review or merge action that must be attributed to them in GitHub, Stoneforge prompts them to link GitHub OAuth and then returns them to the action.
- Review Agent outcomes should be posted back to GitHub as PR reviews while remaining canonical Stoneforge Review Outcomes. Review Approved maps to a GitHub approve review and Change Request maps to a GitHub request-changes review, with the provider review linked back to the Stoneforge Review Outcome, Assignment, and Session.
- Review Agent GitHub PR reviews may satisfy GitHub-side required review rules when GitHub branch protection/rulesets accept the GitHub App or bot review. Stoneforge policy remains canonical for Stoneforge merge readiness. If GitHub requires a human, CODEOWNER, team, or latest-push eligible reviewer that the Review Agent cannot satisfy, Stoneforge should show that as a provider-side merge blocker rather than treating the agent review as sufficient.
- Review Agent dispatch should wait for required Provider Checks to pass or fail by default. If required checks fail, repair triggers before Review Agent dispatch. If a missing required check becomes a policy blocker, Review Agent dispatch should not run because merge is blocked by configuration. Policy may allow early Review Agent dispatch before required verification completes, but the default should conserve review work and include required verification results as review context.
- Workspace policy owns the required Provider Check list for Verification Runs. GitHub branch protection or rulesets may seed defaults but are not the source of truth.
- Stoneforge should not author CI/CD workflows or manage GitHub branch protection/rulesets in the first slice. It may let Workspace policy mark observed GitHub checks/statuses as required for Stoneforge merge readiness, with onboarding suggestions seeded from GitHub branch protection or rulesets when available. GitHub remains the place where CI/CD workflows and provider-side required checks are configured.
- The supervised preset should default to requiring at least one passing Provider Check for code-changing merge, but Workspaces may explicitly disable that requirement.
- If a MergeRequest has no observed Provider Checks while policy requires one, the UI should make the blocker obvious and offer a short disable flow with a warning and confirmation for authorized users.
- Required Provider Checks should be satisfied only by explicit provider success by default: GitHub check-run `success` or commit status `success`. Pending/running/missing/stale observations keep Verification pending or stale. Terminal non-success states fail required checks. `neutral` and `skipped` do not satisfy required checks by default, but Workspace policy may allow either for specific check names.
- Missing required Provider Checks should start as pending while the PR/check observation window is fresh. The observation window should be controlled by a Workspace policy value such as `requiredProviderCheckMissingTimeoutMinutes`, defaulting to 10 minutes after PR head observation. After that timeout, or immediately after GitHub reports that all current checks are complete and the required check is still missing, a missing required check becomes a policy blocker, not a Repair Trigger. The UI should show the missing check name and offer the existing disable/warning-confirmation path for authorized users.
- `stoneforge/policy` should remain pending while a missing required Provider Check is still inside the fresh observation window. Once the missing check is confirmed as a policy blocker, `stoneforge/policy` should be republished as failing with a clear missing-check reason.
- A terminal failing required Provider Check should automatically create a Repair Trigger for the source Task or Plan MergeRequest. Repair Context should include the failing check name, head SHA, provider URL and log summary when available, and whether this is a first or repeated failure. The default is self-healing repair dispatch; do not add a first-slice human-approval exception based on high-risk or sensitive targets.
- Failed optional Provider Checks should not create Repair Triggers or block merge by default, but they should be reported to a Director Agent for triage. They remain visible MergeRequest/provider context and may inform reviewers or agents. If Workspace policy marks the check required, terminal failure follows the required-check repair path instead.
- Review Agent dispatch should not wait for optional-provider-check Director triage. Optional-check triage runs in parallel; if it finishes first, its reviewer context is included in initial review context, and if it finishes later, it is added to the MergeRequest for human review or subsequent/re-run Review Agent review.
- Optional-provider-check Director triage should be grouped per MergeRequest head SHA. Multiple optional failures on the same head should create or update one unresolved Director triage dispatch with all failed optional checks as context. A new head SHA starts a new triage group.
- Optional-provider-check Director triage outcomes should stay simple and non-blocking: no-op with rationale, add reviewer context, or create non-blocking follow-up Tasks.
- Reviewer context produced by optional-provider-check Director triage should be persisted as first-class MergeRequest reviewer context, not only as Director run output. It should keep provenance to the Director Assignment/Session and the failed optional Provider Checks that caused it.
- Optional-provider-check reviewer context should be Stoneforge-only by default. It appears in the Stoneforge MergeRequest review UI and is supplied to Review Agents and human reviewers there, but Stoneforge should not post GitHub comments for optional-check triage by default.
- Provider-check repair should automatically ingest bounded failure context, not full CI logs by default. Bounded context includes check name, conclusion, timestamps, provider URL, job and step names when available, and a failure excerpt or summary when GitHub exposes it cheaply.
- Stoneforge UI should allow users to review full CI logs without leaving Stoneforge only when those logs can be fetched through documented GitHub APIs. GitHub Actions workflow/job logs are the required supported path; other CI provider logs are supported only if GitHub exposes retrievable log data for that check/status without a provider-specific integration. Stoneforge may cache fetched logs with retention limits, but should not eagerly store every full log by default.
- First-slice CI log support should not depend on undocumented GitHub live-log streaming behavior. For in-progress checks, Stoneforge may refresh observed job, step, and check metadata and then fetch logs when GitHub exposes downloadable logs.
- Repair Agents should receive bounded excerpts or summaries by default; they may request full logs through the controlled provider-log access path when needed.
- Agents should be able to run GitHub Actions locally through the `act` CLI inside the resolved Runtime when the Runtime supports it, so they can verify likely CI fixes before pushing repair commits.
- Local `act` verification should default to the failed required Provider Check's corresponding workflow or job when the mapping is clear. If the mapping is unclear, agents should run the smallest relevant workflow or job based on changed files and failure context. Agents should not run every workflow by default, and Stoneforge should not require a dedicated task-progress entry solely to explain local verification coverage.
- Managed Runtimes should include `act` by default when Docker/container execution is available. Customer-managed Runtimes should report whether `act` is installed and usable as a Runtime capability. If `act` is unavailable, agents should fall back to project-local tests and rely on GitHub Actions rerunning after push.
- Local `act` verification should use the same low-risk preview/dev secret boundary as previews by default. Any secrets injected into local Actions verification must be clearly marked as agent-observable and may appear in agent-visible logs; production credentials must not be injected. If a workflow needs secrets that are unavailable or not approved for preview/dev use, the agent should treat local Actions verification as partial and rely on GitHub Actions after push for the unavailable portions.
- Local `act` verification is best-effort and should not promise parity with GitHub-hosted runners, service containers, matrix behavior, hosted runner images, or every GitHub Actions feature. If local `act` behavior is unsupported or divergent, agents should fall back to project-local tests and rely on GitHub Actions after push.
- Local `act` output should remain session-local agent working context only. Stoneforge should not create UI artifacts, lineage records, repair-context items, Provider Checks, Verification Runs, or GitHub checks/statuses from local `act` results. Only GitHub-observed checks can satisfy required Provider Checks.
- Stoneforge should not expose GitHub Actions rerun, workflow dispatch, or arbitrary remote workflow execution controls to agents or humans in the first slice. GitHub Actions should rerun through the normal provider behavior when Stoneforge pushes repair commits to the task or plan branch.
- The first-slice UI direction is already designed in the `reference/smithy-next` prototype and should be used as the product reference, not as active implementation code.
- The `reference/smithy-next` prototype should be treated as the final-goal reference for all initial page designs and workflows. Any modifications, additions, or removals from that direction require explicit user/product approval.
- The first-slice PRD should require workflow and interaction fidelity to `reference/smithy-next`, not pixel-perfect implementation. The prototype is the approved target for information architecture, page coverage, primary workflows, navigation shape, and review/diff interaction direction. Visual polish should be high and consistent, but exact styling, spacing, component internals, and implementation details may change for the active V2 app unless the change alters workflow, page coverage, or user-visible behavior.
- The first-slice PRD should require a practical accessibility baseline: semantic HTML where applicable, keyboard navigability for primary workflows, visible focus states, accessible labels for controls/icons, readable contrast, and screen-reader-friendly status text for blockers, review state, verification, and progress. A full formal WCAG audit, advanced screen-reader workflow certification, localization, and comprehensive keyboard shortcut customization are out of scope.
- First-slice UI scope must include operator surfaces for Automations, Agents, Runtimes, RoleDefinitions, Documents, Tasks, Plans, MergeRequests, Sessions, Workspaces, policy blockers, audit/lineage, and human intervention.
- First-slice UI should include pragmatic searchable and filterable list views for Tasks, Plans, MergeRequests, Sessions, Automations, AutomationRuns, Agents, Runtimes, RoleDefinitions, Documents, and Workspaces using obvious fields such as state, status, owner or actor, linked object, branch, PR, risk, tags, updated time, and blocker presence. This supports operator triage and visibility, not advanced analytics, saved queries, custom dashboards, or cross-workspace reporting.
- First-slice product-facing observability should stay limited to operator workflow visibility: Execution Lineage, Audit Trail, Session transcripts/logs, AutomationRun history, Host/Runtime health, Verification Run/check status, policy blockers, in-app Notification Surfaces, and Operational List filters. Engineering telemetry with OpenTelemetry is required for implementation diagnostics, but metrics dashboards, tracing UI, performance analytics, cost analytics, usage analytics, and Workspace/Org reporting are out of scope.
- First-slice billing/cost visibility should include only configuration that helps control cost: Agent concurrency limits, model/provider selection, Runtime selection, and maybe basic current active Assignment/Session counts. Do not require token accounting, per-task cost reporting, budget alerts, invoices, chargeback, pricing plans, or model-cost analytics. Those belong to a later cost-management slice.
- Diff-heavy MergeRequest review is in scope for the first slice as designed in the UI prototype, because many users and teams will still review agent-written code line by line.
- Documents in the first-slice MVP should support durable workspace context CRUD, create/edit/view/version history, linking to Tasks, Plans, Assignments, MergeRequests, Automations, and RoleDefinitions, and explicit selection as agent context.
- First-slice Documents should be Workspace-owned durable context objects with title, workspace-relative document path or filename, body/content, content type, category/tags, version history, creator/editor attribution, linked workflow objects, visibility according to Workspace policy, and explicit agent-context selection metadata.
- Documents should be organized into at least two first-slice libraries: a standard Workspace documentation library rooted at the Documentation Directory, and a separate system-managed operational docs library for Stoneforge-maintained operational Documents.
- The Documents UI should show the Workspace Documentation Library by default and expose the Operational Docs Library through a clear library selector or filter such as "Operational/System." Operational docs should be visible but separated, not hidden behind a deep admin-only path.
- Operational Docs Library Documents should default to Stoneforge-only source mode in the first slice. They are Stoneforge operational metadata, not team documentation intended to live in the repository docs root. Later slices may allow export or repo-backed operational docs, but the first slice should avoid polluting repository docs and creating unnecessary PR churn.
- Document visibility and access should inherit Workspace policy in the first slice. Document-specific ACLs are out of scope. Sensitive operational knowledge that requires different access belongs in a separate Workspace or behind future document-level controls, not an ad hoc first-slice exception.
- First-slice Documents should support Markdown/plain text as the primary editable type and URL/reference documents for external resources. Rich binary uploads, rendered PDF management, Google Docs import, and complex block-editor documents are out of scope unless the approved UI prototype requires them.
- URL/reference Documents may participate in the Documentation Directory and Level 2/Level 3 Documentation Methodology like other Documents, but their Stoneforge content should be metadata plus optional human- or agent-written summaries/notes. The first slice should not automatically crawl or ingest arbitrary external pages by default. Agents may open referenced URLs only when their RoleDefinition and tool access allow it, and durable knowledge learned from an external source should be summarized into a normal Document when it matters long term.
- External knowledge should become a normal Markdown Document when the team will rely on that knowledge without always revisiting the external source. That Document should include the summary, rationale, and source link. Use a URL/reference Document when the external page itself remains the thing to consult and Stoneforge only needs a pointer plus notes.
- Documents may be Stoneforge-only, repo-backed, or mirrored according to Workspace policy. The default for a Workspace linked to a repository should be mirrored docs, with the repository docs folder selected during repository onboarding.
- Stoneforge Documents are the Workspace context surface used by agents and UI. Repository docs files are source-controlled artifacts when the team wants durable docs in Git. For repo-backed or mirrored docs, docs-drift PRs change repository files, and Stoneforge updates or reindexes corresponding Document objects from the merged source.
- Document version history should be source-mode aware. Stoneforge-only Documents need Stoneforge-owned content version history. Repo-backed and mirrored Documents derive content history from Git commits and PRs, while Stoneforge records indexing events, metadata changes, category/tag changes, link-resolution changes, and agent-context selection events.
- Repository onboarding for mirrored docs should auto-detect common documentation paths such as `docs/`, `documentation/`, `.github/`, and root Markdown docs, then ask the user to confirm or choose one primary docs root. If no docs folder exists, onboarding should offer to create `docs/` with the Documentation Directory. Multiple docs roots are out of scope for the first slice.
- In mirrored mode, repository PR merge is the authoritative write path. Stoneforge UI edits should create a new docs PR or add a commit to an existing docs PR, and the mirrored Document object should import those edits only after merge.
- The Documents UI may let users show or hide pending PR changes, or select a branch/docs PR context to view and edit Documents with pending changes. Conflicting pending changes should resolve through normal source-control merge conflict handling rather than a separate document conflict system.
- Documents should use one canonical internal linking method across all source modes: normal relative Markdown file links. In mirrored or repo-backed modes these resolve to files under the primary docs root; in Stoneforge-only mode they resolve through the Document's workspace-relative document path or filename. The Documents UI should make these links clickable and may offer autocomplete/link insertion, but users should not need to hand-author Stoneforge IDs or slugs.
- Documents should use a lightweight taxonomy: one optional primary system category, plus many freeform Workspace-scoped tags. First-slice system categories should support UI and search facets without creating a nested taxonomy tree, with initial categories such as `spec`, `runbook`, `design`, `decision`, `review-note`, and `reference`.
- Document search should support keyword/full-text search across title and body, with filters for category, tags, content type, linked workflow object, creator/editor, and updated time. Results should make linked Documents easy to traverse.
- Document full-text ranking should use BM25 in both supported local and production database modes: SQLite through its native full-text search path and Postgres through the `pg_textsearch` extension, so search behavior stays reasonably consistent across development and deployment.
- Semantic/vector Document search is out of scope for the first slice unless it becomes trivial to add without distracting from core workflow search.
- Agents should receive Documents through explicit context selection first: linked Documents on the Task, Plan, MergeRequest, or Automation; Documents attached by a human; and Documents selected by the Director during decomposition. Broad Workspace document context should not be injected into every Session by default.
- Agents may use Document Search through the Agent Command Surface, but search should not be the only discovery path. When agents use a searched or browsed Document as material context, the selected Document should be cited or linked in execution lineage.
- Document browsing and Document Search should be distinct agent behaviors. Browsing is an open-ended process where the agent reads full Documents and traverses relative links as it sees fit, usually starting from the Documentation Directory. Search is a targeted process for finding a specific topic, concept, keyword, or filtered set of Documents.
- Document lineage should have two tiers. Stoneforge should automatically record read, search, and browse events for observability, and the Agent Command Surface should also let agents explicitly mark Documents as materially used when they influenced the work. Agents should mark a Document as materially used immediately after relying on it for a decision or implementation direction, not defer that marker until documentation updates. Materially-used Documents are decision/implementation lineage; documentation Acceptance Criteria are a separate requirement about updating relevant Documents or recording why no documentation change is needed.
- Materially-used Document markers should require a short reason that explains why the Document mattered, such as "used routing constraints from runtime architecture" or "confirmed review policy terminology." The reason should be visible to Review Agents and humans inspecting the Task or MergeRequest, but it should stay lightweight and not become a report.
- Agents may also mark materially-used Code References when code files or specific line ranges influenced a decision, implementation direction, or docs-drift fix. This is especially useful for Docs Drift Automation, where the relevant evidence often comes from code rather than existing Documents.
- Agent use of Documents should be mandatory in bounded workflows. Directors should consult the Documentation Directory when decomposing non-trivial or concept-changing work. Workers should consult linked or selected Documents and the Documentation Directory when the Task has documentation Acceptance Criteria or touches documented concepts. Review Agents should check relevant Documents when validating documentation Acceptance Criteria or reviewing concept changes.
- Agents should not be forced to read every linked Document when a Document is clearly irrelevant. They should record which Documents materially informed the work in lineage.
- Each Workspace should maintain a top-level Documentation Directory Document as a table of contents for the Workspace's reusable documentation. It should link to Level 2 entry point Documents and include short descriptions and keyword lists so humans and agents can browse without relying only on search.
- The Documentation Directory should be a hybrid managed/editable Document. Users and agents may edit it like a normal Document, but Stoneforge recognizes it as the Workspace's managed entry point and provides structured assistance such as link validation, missing Level 2 entry point warnings, stale keyword or description warnings, and special Docs Drift Automation attention. It should not be fully generated and overwritten.
- First-slice Documents should follow a three-level Documentation Methodology: Level 1 is the Documentation Directory, Level 2 Documents are entry points with high-level knowledge across topics and links to Level 3 Documents, and Level 3 Documents are focused deep dives on specific concepts. The goal is to keep important documentation reachable within three steps without overloading any one Document with irrelevant context.
- The Documentation Methodology also supports Code References as a Layer 4 reference target. Code is the ultimate source of truth; Documents exist to make building context on the codebase more efficient and to explain why decisions were made. Any Document level may link directly to code files or line ranges when that is clearer or more efficient than duplicating the concept in prose.
- Code References in Document content should use repository-relative file paths with optional line anchors or ranges for readability and portability. Stoneforge lineage should resolve materially-used Code References to the relevant commit SHA when recorded, preserving a stable review/audit target while the UI displays the friendly path and line range.
- Code References should be lightly validated in the first slice. Stoneforge should validate that referenced files exist on the relevant branch and warn on missing paths or obviously stale line anchors/ranges. It should not require perfect semantic validation that referenced code still proves the documented claim. Docs Drift Automation may flag suspected stale Code References during periodic checks.
- First-slice Code References should be limited to the Workspace primary repository. Cross-repository code references should be represented as URL/reference Documents or ordinary external links until cross-repo validation, commit resolution, permissions, and review/audit scope are designed.
- The primary first-slice docs-drift control should be task-local: Tasks should include Acceptance Criteria requiring documentation updates when their code or concept changes add, remove, or change durable knowledge. The task-completing agent should update the relevant Documents using the context it accumulated during implementation.
- Every code-changing or concept-changing Task should include a default documentation Acceptance Criteria such as "Update relevant Documents or record why no documentation change is needed." Pure investigation, review, or administrative Tasks may omit it when they do not affect durable code or product concepts.
- Workers satisfy the documentation Acceptance Criteria either by updating relevant Documents or by recording a short no-docs-needed rationale.
- Review Agents should verify that documentation-update Acceptance Criteria were satisfied when reviewing task work. This avoids spending a separate docs-updater Automation's tokens rediscovering context the implementing agent already had.
- Platform-created default RoleDefinitions should instruct Director, Worker, and Reviewer workflows to use and maintain the Documentation Directory and three-level Documentation Methodology when creating, selecting, updating, or reviewing Documents.
- The platform should provide two default docs-drift and docs-quality Automations, neither of which is a per-code-change updater. A daily scoped Docs Drift Automation should run every configurable X hours, defaulting to every 24 hours overnight, and inspect changed files since the last drift run, open/unmerged docs-drift PRs, existing Code References, and docs-linked areas from the Documentation Directory. A weekly deep Docs Drift Automation should periodically assess broader codebase/documentation alignment.
- Deep docs-drift scanning may split the codebase into sections and dispatch multiple agents, each responsible for one section, to avoid context overload and quality decay from one agent attempting to assess the entire codebase at once.
- Weekly deep Docs Drift Automation should use a plan-shaped workflow. It first dispatches a Director-style agent run to inspect repository structure and Documentation Directory topology, create an appropriate section split, and create a Plan with one docs-drift Task per section. Normal scheduler dispatch then handles those Tasks, with each Worker responsible for its section. Code-changing or docs-changing outputs follow the standard task PR flow and aggregate into a single plan branch/plan PR by default.
- Weekly deep docs-drift Plans follow the same Workspace Plan activation policy as all other Plans. There should not be a docs-specific human activation policy: if the Workspace requires human activation for Plans, the deep-scan Plan waits; if it does not, the deep-scan Plan may proceed under normal scheduler and policy rules.
- The weekly deep Docs Drift Director should create and maintain a stable docs-drift section map Document in Stoneforge. The Docs Drift Section Map belongs in the separate system-managed operational docs library, not in the standard Documentation Directory library by default. The Director should reuse the stable section map when possible and revise it when repository structure or Documentation Directory topology changes. A separate section-boundary change rationale is not required because normal agent Session lineage already captures the Director's reasoning.
- The Docs Drift Section Map should be an authorized human-editable Markdown Document without schema validation in the first slice. Human edits are versioned and visible in normal Document lineage. The next Director sectioning run interprets the map and may preserve, repair, or revise it when it is unclear, stale, or inconsistent with repository or documentation topology.
- When a human manually creates or edits a Document, the UI should offer an optional manual trigger for a docs-drift AutomationRun scoped to that specific change.
- Docs Drift Automations should dispatch agents to inspect and fix documentation issues they find, because those agents have already built the context needed to understand the issue. They should accumulate fixes on a dedicated docs-drift branch and draft PR, then mark the PR ready when the Automation completes.
- Docs Drift Automation should record materially-used Code References when code evidence drives a docs-drift finding or fix. Materially-used Document markers are still useful when existing Documents influenced the fix, but they are not the primary evidence path for code-to-docs drift.
- Docs-drift Tasks should be docs-only in the first slice. If a docs-drift agent finds code issues while reviewing drift, it should report them to a Director for triage or follow-up Task creation, not patch code inside the docs-drift Task.
- Docs-drift Tasks may create new Documents when the Documentation Methodology calls for it. A section Worker may create missing Level 2 or Level 3 Documents, update the Documentation Directory, add Code References, and update relevant links. In mirrored mode, new or edited Documents still flow through the docs PR and standard review process.
- Daily scoped Docs Drift Automation should use one docs-drift PR per run. Weekly deep docs drift should aggregate section task PRs into one plan branch and one plan PR by default, matching the plan-shaped workflow. If a section produces risky or unrelated doc changes, the Director or policy can split it into a follow-up Plan or Task rather than mixing it into the shared PR.
- Docs-drift PRs should use the same MergeRequest framework as code PRs, while allowing policy to distinguish docs-only changes. Defaults should require Review Agent review for docs-only PRs. Human Review follows the Workspace policy preset and any docs-only exemptions: supervised may require Human Review unless exempted; autopilot may merge after agent review plus observed Provider Checks, Mergeability, and policy checks when applicable.
- Stoneforge should create a default docs-focused Reviewer RoleDefinition for docs-only and docs-drift PRs. It remains in the reviewer role category and uses the same Review Outcome model, but its prompt/checklist focuses on clarity, correctness, link and Code Reference quality, Documentation Methodology fit, and whether any doc deletion is justified. Workspaces can edit or delete it like other default RoleDefinitions.
- Code PRs that include documentation changes should keep the normal code Review Agent as the sole automatic Review Agent. Stoneforge should not add a second docs-focused review automatically for mixed code/docs PRs; instead, normal code Reviewer RoleDefinitions should include docs-focused review pointers for documentation Acceptance Criteria, Document Link quality, Code Reference quality, and no-docs-needed rationales.
- Docs-only Human Review exemptions should be explicit Workspace policy toggles. `supervised` should default to Human Review for docs-only PRs like other PRs unless the Workspace enables a docs-only exemption. `autopilot` should default to agent-only review for docs-only PRs, subject to applicable Provider Checks, Mergeability, and policy checks.
- Docs-only classification should be determined by changed file extensions, not by docs-root path. A PR is docs-only when every changed file is a recognized Markdown or text documentation file, such as `README.md`, `.md`, `.mdx`, `.txt`, or other configured text-doc extensions. Docs-drift Tasks and PRs may still be tagged as docs-only intent, but final docs-only policy applies only after extension-based changed-file verification.
- The docs-only extension set should be Workspace-configurable with a small default list such as `.md`, `.mdx`, `.txt`, and `.rst`. This keeps the first-slice default simple while supporting teams that use other text documentation formats.
- Config-like files such as `.json`, `.yaml`, and `.yml` should not count as docs-only by default because they may affect runtime behavior. Workspaces may deliberately add structured documentation extensions if needed.
- Docs-only classification should treat creates, edits, and deletions the same way: if every changed file has a recognized docs-only extension, the PR remains docs-only. Deleting docs may still be risky, but review policy and Review Agent output handle that risk rather than reclassifying it as code.
- Changing the Workspace docs-only extension set should warn when open MergeRequests would be reclassified and require confirmation from an authorized user. Open MergeRequests are re-evaluated under the new policy; terminal or merged MergeRequests keep their historical classification.
- Stoneforge should not define, publish, or run a dedicated docs-specific GitHub check/status for docs-only PRs in the first slice. It publishes the normal Stoneforge Policy Check and observes GitHub/provider checks as Provider Checks inside Verification Runs. Docs quality, Document Link, and Code Reference findings belong in Stoneforge MergeRequest UI and Review Agent output, not a separate GitHub check.
- Stoneforge should publish only the normal `stoneforge/policy` GitHub check/status in the first slice. Provider Checks come from GitHub Actions or other GitHub-integrated CI providers; local `act` stays inside the agent session and does not produce a provider-visible status.
- Non-GitHub CI providers are supported only through GitHub's checks/statuses surface in the first slice. Stoneforge should treat any observed GitHub check/status as a Provider Check regardless of whether it originated from GitHub Actions, Buildkite, CircleCI, Jenkins, or another provider. Full in-app logs for those providers are supported only when available through GitHub APIs; otherwise Stoneforge shows the GitHub-observed status/details/URL and leaves provider-specific log viewing to the CI provider. Direct CI provider integrations, provider-specific log APIs, and provider-specific rerun controls are out of scope.
- Docs Drift Automation must account for in-flight docs-drift PRs. If a prior docs-drift PR is still open and unmerged, later runs should treat its proposed changes as pending context, avoid repeating the same fixes, and either skip overlapping findings, append to the existing docs-drift branch when safe, or create only non-overlapping follow-up work according to policy.
- Once the docs-drift PR is ready, the standard MergeRequest flow applies: Review Agent review, Verification Run observation if relevant, automatic merge or Required Human Review according to Workspace policy.
- Workspace policy should provide an easy way to disable the Documentation Methodology. When disabled, default RoleDefinition documentation-update instructions are removed or hidden, the default documentation Acceptance Criteria is not added to Tasks, the periodic docs-drift Automation is disabled, and manual docs-drift trigger suggestions on Document edits are disabled.
- Documents are not task progress records, transcripts, or hidden prompt memory. They hold reusable context worth preserving beyond one Assignment.
- Documents should not require Google Docs-style real-time coauthoring, rich redlines, or document-specific permissions beyond Workspace policy in the first slice.
- Sessions should support live operator steering, not just read-only observation. The first slice should include session list/detail, live status, event timeline, transcript/log viewing, checkpoint visibility, message-to-session where the adapter supports it, cancel/force-stop, and resume from checkpoint.
- Any authorized human team member may send a message to steer an active or continuable Session when the adapter supports it. Provider-facing agent chat may flatten those messages to `User`; Stoneforge should still record the authenticated human sender in lineage/audit.
- Code inspection and lightweight manual edits are in scope as an intervention path across Workspace branches and linked workflow objects.
- The UI may include a simple Monaco-style editor integrated across tasks, MergeRequests, diffs, and related views for following code paths, inspecting files/diffs, making edits, and committing changes. The editor should not artificially restrict humans to Task, Plan, or docs PR branches; authorized users may edit files on any branch and commit the changes through the UI when provider permissions and branch protections allow it.
- Human editor commits should be attributed to the authenticated human where provider identity allows, shown in Stoneforge lineage when linked to Workspace objects, and should trigger the same provider observation and policy re-evaluation as other commits when they affect an observed MergeRequest or workflow branch.
- Editor commits that are not linked to an active Task, Plan, docs PR, or MergeRequest are allowed as repository branch edits when the authenticated human has provider permission. Record provider attribution and a Stoneforge lineage/audit event if the branch belongs to the Workspace, then observe any resulting provider checks or PR changes if a PR exists. Do not automatically create a Task or Review workflow for unlinked editor commits.
- If an unlinked editor commit lands on a branch with an observed MergeRequest, normal Verification Run, Required Review freshness, Mergeability, and `stoneforge/policy` re-evaluation apply. Direct target-branch commits are governed by GitHub permissions and branch protection, not by a synthetic Stoneforge merge workflow.
- The editor may offer agent-assisted commit message generation. Generated commit messages are human-reviewed suggestions and do not make the commit an agent-authored commit.
- The editor is low-visibility supporting infrastructure, not a core workflow or a general-purpose IDE replacement in the first slice.
- CI/CD should be verification-observation first: list GitHub/provider checks, show run/job details and logs when available from the provider, connect failures to repair handoff, allow local `act` verification inside capable Runtimes, and feed Verification Runs. Stoneforge should not author or host CI workflows natively, manage GitHub branch protection, become the provider-side required-check configuration surface, expose remote GitHub Actions rerun/dispatch controls, or depend on undocumented live-log streaming in the first slice.
- Preview environments must be supported for all first-slice Runtimes and Hosts, allowing users to spin up and interact with previews from task or plan-specific branches.
- Provider-supplied preview URLs/statuses are optional for the first slice, but may be included if easy.
- Preview is not deployment promotion, rollback, or general environment management.
- An acceptable preview environment launches or connects to the app process for a task or plan branch, exposes a reachable URL through the runtime/host path, reports basic lifecycle status (`starting`, `ready`, `failed`, `stopped`), allows stop/restart, and links to the relevant Task, Plan, or MergeRequest.
- First-slice previews do not require scaling infrastructure, custom domains, or production-like environment management.
- First-slice secret use should default to capability-based access rather than raw secret exposure to LLM-visible agent environments.
- Agent workflows that need provider credentials should use Stoneforge-controlled APIs, CLIs, tool calls, or proxies such as GitHub operations, registry/package installation, artifact access, runtime provider operations, and webhook signing. The agent receives a sanitized result, not the secret value.
- The first-slice Agent Command Surface should expose only policy-checked, lineage/audit-recorded actions needed during assigned work: reading assigned context, updating task progress and Checkpoints, adding task-local todos, reporting outcomes, requesting escalation, starting/stopping/inspecting previews, and invoking secret-backed capabilities through Stoneforge proxies.
- The Agent Command Surface must not let agents bypass Scheduler or policy, change Acceptance Criteria, mutate unrelated workflow objects, access raw secrets, manage users or policy, or directly merge/administer provider resources outside controlled Stoneforge operations.
- The Agent Command Surface is specifically the Stoneforge control-plane/data API/CLI/tool surface. It is different from general agent tool use such as file read/write tools, shell access, provider-native coding tools, or custom tools supplied to agents.
- Agents should authenticate to the Agent Command Surface with a short-lived Session Command Credential minted by Stoneforge for the Assignment/Session. It should be scoped to the Workspace, target Task or MergeRequest, resolved branch/worktree/runtime, and command categories allowed by effective policy and target context.
- RoleDefinitions should not request or restrict Agent Command Surface categories in the first slice. All RoleDefinitions should receive the same command surface shape until useful separation boundaries are known.
- RoleDefinitions may still constrain general agent tools, file/shell/tool access, skills, and lifecycle hooks.
- Session Command Credentials should expire or rotate, be revocable on cancel/stop, and must not be human user credentials or provider secrets. Agent command lineage should record the selected Agent and Session plus the service-actor path, adding human lineage only when a human steers the Session.
- For code-changing Assignments, agents may use ordinary `git` inside the resolved Runtime worktree and branch with short-lived GitHub App installation credentials. GitHub installation access tokens can be scoped to repositories and permissions, but not to a single branch as a credential-level sandbox.
- Stoneforge should use the minimum necessary GitHub App repository permissions, short token lifetime, branch naming/lease ownership, GitHub branch protection or rulesets where configured, provider event observation, and Agent Command Surface helpers for common controlled operations like push/open MergeRequest.
- Merge, branch protection bypass, provider administration, and cross-target branch mutation must go through Stoneforge-controlled policy checks rather than raw agent `git` authority.
- If an agent mutates an unauthorized GitHub branch, Stoneforge should treat it as a recoverable policy violation and Branch Health/Repair Trigger. The default recovery should follow the normal repair-trigger path: attach Repair Context to the Task, create a new task-owned repair Assignment and Session, undo unauthorized branch mutations, and decide whether any useful changes should be moved by cherry-pick, rebase, or equivalent onto the originally scoped branch. Human escalation should be reserved for failed/unsafe recovery or policy-defined sensitive cases.
- Raw secret injection into an agent-visible shell or filesystem should be an explicit high-risk escape hatch, if supported at all.
- Preview environment variables are a special case: app code can intentionally print them to a page, API route, or log where the agent can read them, so Stoneforge should not present preview env injection as a hard security boundary.
- Preview env vars should be clearly marked as low-risk/dev-preview secrets that may be exposed to an adversarial or compromised LLM agent. It is still useful to provide Stoneforge-controlled API, CLI, or tool commands for starting, stopping, and inspecting preview processes, but split-process/container separation must not be treated as sufficient protection for sensitive secrets.
- Stoneforge should not claim it can reliably infer whether a secret value is production or preview-safe. Preview-safety is primarily user-declared at configuration time, with clear UI warnings and optional provider-specific heuristics only where the distinction is obvious, such as known test/live key prefixes.
- Each Preview Secret must be explicitly marked as allowed for preview/agent-observable use. When a user configures multiple Preview Secrets in one action, a single clear approval can cover the whole batch rather than requiring one approval per key.
- Customer-managed Hosts should register to a Workspace with a name, host type, reachability/heartbeat state, supported Runtime labels/capabilities, capacity limits, and a scoped enrollment token.
- Customer-managed Hosts connect outbound to Stoneforge, advertise capacity, receive assignment leases, and report health. The first slice should support reconnect, disable, and remove flows, but not fleet autoscaling or complex host scheduling.
- Disabling a Host should stop new dispatch/leases to that Host and mark queued or ready work blocked if no acceptable placement remains, while allowing running Assignments to continue when the Host remains connected.
- Removing or force-disconnecting a Host with running work should require confirmation and must either block until the work is stopped or explicitly stop/mark those Assignments interrupted. Running work must not be silently orphaned.
- Unexpected Host disconnect should first create an uncertain contact-loss condition, not immediate task or Assignment failure.
- On Host reconnect, Stoneforge should reconcile running Sessions: if the prior Session is still healthy, it returns to normal without resume; if the Host-wide outage stopped the process, the Session is marked interrupted/crashed and the Assignment follows resume or redispatch policy.
- Stoneforge control plane owns the authoritative Session and Assignment state transition after reconnect reconciliation.
- The Host Agent should report live process/session inventory, terminal outcomes it observed, logs/checkpoints since the last heartbeat, and any unknown gaps; the provider adapter should validate provider-specific Session identity and resumability where applicable.
- Assignment failure should occur only after retry/resume policy is exhausted or an operator explicitly abandons the work.
- First-slice customer-managed Runtimes should remain statically bound to one specific Host plus one execution mode, matching the current UI prototype. The Scheduler checks the selected Host's health and capacity at dispatch, but does not dynamically choose among interchangeable Hosts.
- Runtime disable or capacity/config edits should affect future placement only by default: stop new dispatch to that Runtime and block queued or ready work if no acceptable Runtime remains.
- Running Assignments keep their resolved Runtime and continue unless an operator or policy explicitly stops them.
- Destructive Runtime removal should require confirmation when running or queued work depends on it, and queued work should receive placement blockers and repair actions rather than silent cancellation.
- Assignment capacity leases should reserve the selected Runtime/Host or provider capacity, plus any configured Agent throttle, for the full Assignment lifecycle. Child Sessions run under that same reservation.
- If a Session ends recoverably, enters Contact Loss, or the Assignment moves to `resume_pending`, the Assignment should keep its capacity reservation so it can resume without competing for placement again.
- Assignment capacity reservations are released on Assignment success, cancellation, escalation, explicit abandonment, or after a configurable stale timeout for unreachable/lost work.
- Session does not always mean a live network connection. For most providers, a Session is a durable provider execution context or provider-side conversation record that can be continued later by provider Session ID; connection/reconnect semantics apply only when the adapter/provider requires them, such as Codex App Server.
- Adapter Session connectivity capability should use the simple values `connectionless` and `connectionful`. `connectionless` means gaps between prompts are normal and continuation uses provider Session ID; `connectionful` means observation/control depends on an active connection and can enter Contact Loss.
- For `connectionless` adapters, one Stoneforge Session should correspond to one provider Session ID. Continuation prompts are Session events/messages, not new Stoneforge Sessions.
- A new Stoneforge Session is created only when Stoneforge switches to a different provider Session ID, recovers from a non-continuable failure, or deliberately starts a replacement execution context under the same Assignment.
- Provider Session IDs are scoped external identifiers, not globally canonical Stoneforge IDs. Store them with Workspace, adapter/provider, provider account or installation when relevant, and provider Session ID; Stoneforge `SessionId` remains the global canonical identifier.
- For `connectionful` Sessions, Contact Loss pauses both operator observation/control and autonomous Assignment progress. Stoneforge should not issue prompts or commands while Session state is unknown.
- After reconnect reconciliation, a healthy Session resumes normal progress; a lost or stopped Session moves the Assignment to `resume_pending` or escalates by policy.
- Later slices may add dynamic Host placement by letting Runtimes define Host capability requirements, but that is not part of the first-slice PRD.
- Agents should not be limited to one static Runtime. An Agent should have a set of acceptable Runtimes, each with an optional numeric priority score. The simple case remains one acceptable Runtime, but this allows fallback or more flexible placement without introducing dynamic Host pools.
- First-slice Runtime selection for an Agent should prefer eligible acceptable Runtimes by numeric priority score. If multiple eligible Runtimes have the same priority, deterministic tie-breaking is sufficient; load-balancing is not required.
- If all acceptable Runtimes for an otherwise eligible Agent are unavailable or at capacity, the Dispatch Intent remains durable and queued. The Scheduler records the placement reason, retries on health/capacity changes or backoff expiry, and escalates only after workspace policy thresholds are crossed.
- The Scheduler must not silently switch to an Agent or Runtime outside the acceptable set.
- RoleDefinitions must not express Agent or Runtime preference. Any RoleDefinition may be combined with any Agent for an Assignment; Agent and Runtime selection are driven by Dispatch Intent constraints, Agent configuration, policy, health, capacity, and Runtime availability.
- Provider/model routing such as Codex vs Claude belongs in Agent tags and required Agent tags on Tasks, Automations, or one-off Dispatch Intent constraints, not in RoleDefinitions.
- First-slice Required Agent Tags are hard eligibility constraints only. Preferred Agent tags and tag scoring are out of scope.
- Stoneforge should not add a separate specific-Agent constraint in the first slice. If an operator wants to pin work to one Agent, they can add an Agent-id-style tag to that Agent and require that tag.
- Stoneforge should automatically create a stable system-managed Agent-id tag for every Agent. It should be visible/selectable in advanced routing UI, while normal workflows use human-readable capability tags.
- System-managed tags such as Agent-id tags are visible/selectable but not user-editable or removable. User-defined tags remain editable according to policy/RBAC.
- If an Agent is deleted, its system-managed Agent-id tag disappears with it. Queued work requiring that tag becomes unplaceable until edited or escalated by policy.
- When a required Agent tag becomes unsatisfiable, Stoneforge should immediately mark affected queued work with an `unplaceable` placement blocker and surface repair actions in the UI. The work is not automatically canceled; escalation or notification follows workspace policy.
- Repair actions for an `unplaceable` required Agent tag include editing required tags, restoring or creating an Agent with that tag, or canceling the work.
- Plan-level required Agent tags act as creation defaults for newly created Tasks, not as live inherited constraints. A Task owns the effective required Agent tags used for dispatch; changing Plan defaults later does not silently mutate existing Tasks unless the user explicitly applies the change.
- Agent tags may be unscoped capability tags or scoped routing tags. Unscoped capability tags are additive hard requirements. Scoped routing tags have one effective value per scope, so Task-level scoped tags override Plan defaults for the same scope instead of stacking into impossible combinations.
- Mutually exclusive routing dimensions such as provider, model family, and Agent-id pinning should use scoped tags. If a Task directly contains conflicting scoped tags in the same scope, Stoneforge should mark routing constraints invalid before dispatch.
- Scoped tag syntax is `scope:value`: anything before the first colon is the scope, and anything after it is the value. Tags without a colon are unscoped additive tags.
- First-slice first-class scoped Agent tag dimensions are `provider`, `model-family`, `model`, and `agent`.
- `provider` selects provider/harness family such as Claude or Codex. `model-family` selects broad model family such as Opus or Sonnet. `model` pins an exact model ID. `agent` pins to the system-managed Agent-id tag.
- First-class `provider`, `model-family`, `model`, and `agent` tags on Agents are system-derived from Agent configuration, visible in Agent creation/edit UI, and non-modifiable by users.
- When an Agent provider, model family, model, or identity changes, Stoneforge updates the corresponding system-derived scoped Agent tags.
- If changing an Agent's provider/model/model-family/identity or disabling an Agent would block queued or ready work, the Agent UI should show the affected work count and require confirmation before saving.
- Authorized Agent edits or disables may still proceed after confirmation, but affected work receives placement blockers and repair actions rather than being silently canceled.
- Running Assignments are not affected by later Agent configuration or system-derived tag changes. They keep the resolved Agent, Runtime, RoleDefinition, and provider Session context they started with.
- Agent edits, disabling, or availability changes use the same baseline rule: they affect future dispatch and queued/ready placement, but do not implicitly mutate, stop, cancel, or migrate running Assignments.
- Stopping, canceling, or migrating running work after an Agent change must be an explicit operator or policy action, not an implicit side effect of editing or disabling Agent configuration.
- Scoped tag validation should happen at creation/edit time and again at dispatch time. UI forms should prevent or warn on invalid same-scope combinations, and dispatch should produce an `invalid_routing_constraints` placement blocker as a safety net.
- Plan-default scoped tags and Task-level scoped tags are merged with smart override behavior: Task-level scoped values override Plan defaults for the same scope, so they do not conflict merely because the Plan default had another value.
- The first-slice persistence model should move toward normalized relational tables early rather than relying on broad JSON snapshots as the long-term production shape.
- Use a typed SQL layer such as Drizzle at the control-plane infrastructure boundary to gain database-level constraints, typed query surfaces, and migration discipline.
- Domain packages must remain independent from SQL drivers and query builders. Package interfaces own domain behavior; the control-plane persistence layer maps domain records to relational tables.
- JSON snapshot storage may remain a dev/test fallback or compatibility bridge, but production-oriented first-slice persistence should prefer normalized tables for core workflow objects, relationships, provider facts, AuditEvents, and Workflow Events.
- Core workflow objects should have normalized current-state tables that answer what is true now.
- AuditEvents and Workflow Events should be immutable append-only records. AuditEvents cover sensitive actions and compliance; Workflow Events cover lifecycle/projection facts such as task readiness changes, dispatch intent creation, Assignment/Session lifecycle, checkpoint creation, MergeRequest opening, Verification Run observation, repair triggers, policy decisions, preview lifecycle changes, and automation runs.
- First-slice human authentication must support clearly marked single-user local mode for OSS/dev and delegated OIDC/OAuth for self-hosted team use.
- Stoneforge must not build custom passwords, MFA, or a full identity provider in the first slice.
- Single-user mode actions should still be attributed to a local human principal so audit and lineage semantics remain intact.
- Self-hosted authenticated users should map into Org/Workspace membership and subject classes.
- First-slice RBAC should follow GitHub's organization and repository permission model, including GitHub-style org owner/admin/member concepts, repository roles such as read, triage, write, maintain, and admin, and repository rules for required PR reviewers and teams.
- Stoneforge should not introduce a separate Approver role. Reviewers either record Review Approved or request changes; there is no separate Approval record distinct from Review Approved.
- For GitHub-backed Workspaces, GitHub is the first-slice source of truth for repository access and human reviewer eligibility. Stoneforge stores projections for UI, policy evaluation, and audit.
- Stoneforge policy may add workflow requirements on top, such as Required Agent Review or Director activation review, but must not contradict GitHub repo permissions or allow a human to review/merge through Stoneforge when GitHub would not allow the equivalent repo action.
- In self-hosted team mode for GitHub-backed Workspaces, delegated OIDC/OAuth authentication proves the Stoneforge user, but repo-equivalent review and merge actions require a linked GitHub identity whose repository permissions satisfy the relevant GitHub-style rule.
- Stoneforge org/workspace membership alone may authorize Stoneforge-only actions such as viewing, task creation, steering, and policy administration, but it must not satisfy Required Human Review or authorize merge requests into GitHub without a linked eligible GitHub identity.
- In local OSS/dev single-user mode, Stoneforge should treat the local human principal as repo admin, operator, and reviewer for the connected Workspace, with UI clearly marking the mode as local/single-user.
- In local OSS/dev single-user mode, Required Human Review can be satisfied by the local human principal unless the Workspace is explicitly connected to GitHub identity and configured to enforce GitHub reviewer eligibility.
- Local single-user actions must still emit audit and lineage with the local human principal.
