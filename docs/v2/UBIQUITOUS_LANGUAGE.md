# Ubiquitous Language

This glossary captures the canonical domain language for Stoneforge V2 work. It is subordinate to the V2 charter in [README.md](README.md) and should evolve with the rest of the V2 docs.

## Product frame

| Term                   | Definition                                                                                                                                                                              | Aliases to avoid                                  |
| ---------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------- |
| **Stoneforge V2**      | A control plane for supervised, agent-driven software-engineering execution.                                                                                                            | Chat app, project tracker, code editor            |
| **Control Plane**      | The Stoneforge system of record for intent, planning, dispatch, policy, execution lineage, review, approval, and recovery.                                                              | Orchestrator app, daemon, editor                  |
| **Engineering Intent** | A human's request, idea, or plan for changing or understanding a codebase.                                                                                                              | Prompt, ticket text, vague ask                    |
| **GitHub Issue Intent** | A GitHub Issue imported into Stoneforge as an engineering-intent source or issue-backed Draft Task according to Workspace policy, with enough synced issue context for Director triage. | Executable task, required task, ticket sync       |
| **Execution Lineage**  | The recorded chain of planning objects, dispatch attempts, Sessions, Review Outcomes, Verification Run observations, Required Review satisfaction, and merge actions for a piece of work. | Timeline, history, logs, transcript               |
| **Workflow Event**     | An immutable non-audit lineage event that records workflow progress, lifecycle changes, provider observations, or projection facts for reconstruction and operator visibility.          | Audit event, log line, timeline item              |
| **Projection**         | A derived read model built from canonical records or append-only events for querying, UI, automation decisions, or summaries.                                                           | Source of truth, cached state, denormalized table |
| **Audit Trail**        | The compliance-oriented subset of Execution Lineage made from required AuditEvents for sensitive actions.                                                                               | Timeline, activity feed, full history             |
| **Human Intervention** | A normal operator action that inspects, steers, resumes, cancels, reviews, approves, or directly repairs work; the broad umbrella for human control inside automated flows.             | Manual fallback, failure mode, escape hatch       |
| **Notification Surface** | In-app Stoneforge UI surfaces that show blocked or action-required work to operators, such as readiness blockers, required human review, failed verification, repair escalation, blocked AutomationRuns, Host or Runtime disconnects, and missing configuration. | Email, Slack, digest, alerting system             |
| **Operational List**   | A searchable and filterable first-slice UI list for operational objects such as Tasks, Plans, MergeRequests, Sessions, Automations, AutomationRuns, Agents, Runtimes, RoleDefinitions, Documents, and Workspaces. | Analytics dashboard, saved query system           |
| **Product Observability** | The first-slice operator-visible evidence surface for understanding workflow state: Execution Lineage, Audit Trail, Session transcripts/logs, AutomationRun history, Host/Runtime health, Verification Run and Provider Check status, policy blockers, Notification Surfaces, and Operational Lists. | Metrics dashboard, tracing UI, analytics suite    |
| **Accessibility Baseline** | The practical first-slice UI accessibility bar: semantic HTML where applicable, keyboard navigation for primary workflows, visible focus states, accessible labels, readable contrast, and screen-reader-friendly status text. | Formal WCAG audit, localization program           |
| **Telemetry Signal**   | A diagnostic OpenTelemetry span, log, or metric emitted by backend execution for observability and debugging.                                                                           | AuditEvent, Workflow Event, source of truth       |
| **Trace Context**      | The propagated OpenTelemetry correlation context that links related Telemetry Signals across backend, host-agent, adapter, and webhook boundaries.                                      | Audit context, task context                       |

## People and actors

| Term                 | Definition                                                                                                                             | Aliases to avoid            |
| -------------------- | -------------------------------------------------------------------------------------------------------------------------------------- | --------------------------- |
| **Human User**       | A person authenticated into Stoneforge through an upstream identity system.                                                            | Account, login              |
| **Engineering Lead** | A primary user who supervises engineering work across tasks, plans, agents, and reviews.                                               | Manager, project owner      |
| **Platform Lead**    | A primary user who owns execution capabilities, integrations, policy, and operational reliability.                                     | Infra owner, admin          |
| **Operator**         | A human who creates, dispatches, steers, resumes, cancels, or escalates work inside a Workspace.                                       | Dispatcher, controller      |
| **Human Reviewer**   | A human authorized to inspect a MergeRequest and record Review Approved or a Change Request.                                           | QA, separate approval role |
| **Service Actor**    | A non-human actor such as the scheduler, GitHub App, host agent, provider adapter, or webhook handler acting within Stoneforge policy. | Bot, system user            |

## Tenant and operational boundaries

| Term                        | Definition                                                                                                                                                           | Aliases to avoid                       |
| --------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------- |
| **Org**                     | The top-level tenant, membership, identity, and administration boundary.                                                                                             | Organization account, company          |
| **Workspace**               | The primary operational boundary for one repository, its policies, execution capabilities, tasks, plans, documents, and audit partition.                             | Project, repo, tenant                  |
| **Repository**              | The source-control repository linked to a Workspace, with GitHub as the first-slice provider.                                                                        | Codebase, project                      |
| **Provider**                | A generic external service Stoneforge integrates with under policy, such as source control, CI, identity, runtime, notification, or automation services.             | Integration, vendor, adapter           |
| **Source-Control Provider** | A Provider that owns repositories, branches, MergeRequests, source-control checks or statuses, and merge operations.                                                 | Provider, Git host, repository service |
| **GitHub Provider**         | The first-slice Source-Control Provider used for repository access, PRs, checks/statuses, and merge operations.                                                      | GitHub App, GitHub integration         |
| **Policy**                  | The system that determines what may happen automatically and what requires review before merge or other sensitive workflow actions.                                  | Permissions, settings, rules           |
| **Policy Preset**           | A named bundle of workspace policy defaults that supplies a coherent automation and Required Review posture while still allowing specific policy values to be changed. | Mode, profile                          |
| **Policy Configuration Surface** | The first-slice Workspace UI for selecting a Policy Preset, configuring documented first-slice policy values, and showing effective policy/readiness consequences. | Policy editor, rule builder            |
| **Supervised Policy**       | The default preset that allows automated dispatch and review while requiring at least one qualifying Human Review for code-changing merge unless exempted.           | Manual mode, safe mode                 |
| **Autopilot Policy**        | A preset that allows automatic merge when policy, verification, mergeability, and review conditions are satisfied, with agent review required by default.            | Autonomous policy, full auto, hands-off mode |

## Planning and context

| Term                      | Definition                                                                                                                                                                 | Aliases to avoid                                           |
| ------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------- |
| **Task**                  | The primary planning unit for intended work.                                                                                                                               | Job, ticket, run                                           |
| **Issue-Backed Draft Task** | A Draft Task created from a GitHub Issue under Workspace policy before Director triage or human edit makes it dispatchable.                                                | Synced issue, executable issue, GitHub task                |
| **Issue Intake Policy** | Workspace policy that decides how imported GitHub Issues become Stoneforge intent, non-dispatchable draft Tasks, Director triage work, or ready Tasks.                              | Issue sync, webhook rule, GitHub automation                |
| **External Issue**      | A GitHub Issue created by a user who is not trusted by the Workspace policy for automatic Stoneforge intake.                                                               | Public issue, untrusted task                               |
| **Trusted Maintainer Label** | A reserved `stoneforge:*` GitHub Issue label whose Stoneforge intake effect is honored only when the label was applied by a linked trusted maintainer with sufficient GitHub-backed repository permission. | Magic label, automation label                              |
| **Stoneforge Issue Label** | A reserved `stoneforge:*` GitHub Issue label written or interpreted by Stoneforge under policy. Intake command labels such as auto-dispatch and auto-triage can affect intake; status, priority, and complexity labels are Stoneforge-owned projections. | GitHub source of truth, arbitrary label                    |
| **Plan**                  | A coordinated execution-supervision grouping for related tasks.                                                                                                            | Epic, project, second task engine                          |
| **Plan Graph**            | The shaped set of planned tasks and dependency relationships that must be coherent before activation.                                                                      | Task list, roadmap                                         |
| **Dependency**            | A blocking relationship that prevents a task from becoming ready until another task or constraint is satisfied.                                                            | Link, prerequisite note                                    |
| **Acceptance Criteria**   | The concrete conditions that define whether a task's intended work is complete.                                                                                            | Requirements, checklist                                    |
| **Priority**              | A task ordering signal used for planning and scheduler decisions.                                                                                                          | Rank, severity                                             |
| **Document**              | Workspace-owned durable shared context such as specs, runbooks, design notes, and reusable review knowledge, with title, workspace-relative path or filename, content, content type, category/tags, version history, attribution, links, visibility, and explicit agent-context selection metadata. | Hidden memory, task progress note, transcript              |
| **Document Library**      | A top-level grouping for Documents, such as the standard Workspace documentation library or the system-managed operational docs library.                                      | Folder, category, permission boundary                      |
| **Workspace Documentation Library** | The standard Document library rooted at the Documentation Directory and used for team-facing reusable workspace knowledge.                                           | Operational docs, hidden memory                            |
| **Operational Docs Library** | A separate system-managed Document library for Stoneforge-maintained operational Documents such as the Docs Drift Section Map.                                            | Documentation Directory, normal workspace docs             |
| **Document Source Mode**  | Workspace policy for whether Documents are Stoneforge-only, repo-backed, or mirrored with repository docs files.                                                             | Sync implementation detail, storage backend                |
| **Primary Docs Root**     | The single repository documentation root selected for a first-slice Workspace using mirrored or repo-backed Documents.                                                        | Monorepo docs graph, arbitrary file roots                  |
| **Mirrored Document**     | A Document mode where Stoneforge keeps a UI/agent context object and a corresponding repository docs file in sync, with repository changes reviewed through PRs.              | Duplicate doc, cache-only copy                             |
| **Repo-Backed Document**  | A Document mode where the repository docs file is the source-controlled artifact and Stoneforge indexes or presents it as Workspace context.                                  | External attachment, unmanaged file                        |
| **Document Version History** | Source-mode-aware history for Document content and metadata: Stoneforge-owned content history for Stoneforge-only Documents, Git-derived content history for repo-backed and mirrored Documents, and Stoneforge lineage for metadata/indexing/context events. | Generic edit log, transcript history                      |
| **URL/Reference Document** | A lightweight Document that points to an external resource and stores metadata plus optional human- or agent-written summaries and notes, without automatic first-slice crawling or ingestion. | Crawled page, imported external doc                        |
| **Document Link**         | A normal relative Markdown file link from one Workspace Document to another, resolved by Stoneforge to a clickable Document target across mirrored, repo-backed, and Stoneforge-only source modes. | Prose-only reference, external URL, Stoneforge ID link      |
| **Document Category**     | An optional primary system category used to organize and filter Documents, such as `spec`, `runbook`, `design`, `decision`, `review-note`, or `reference`.                  | Folder tree, document type hierarchy                       |
| **Document Tag**          | A freeform Workspace-scoped label applied to Documents for organization, filtering, and search.                                                                              | Category, global taxonomy                                  |
| **Document Search**       | First-slice keyword/full-text search over Document title and body, ranked with BM25 across SQLite and Postgres and filterable by Document metadata and linked workflow objects. | Semantic search, vector memory                             |
| **Document Browsing**     | Open-ended agent or human reading of full Documents and traversal of relative Document Links, usually starting from the Documentation Directory.                             | Keyword search, hidden context injection                   |
| **Materially Used Document** | A Document an agent explicitly marks, immediately after relying on it, as having influenced a decision or implementation direction, with a short reason explaining why it mattered. | Read receipt, incidental document view, docs update marker |
| **Code Reference**        | A repository-relative link from a Document or lineage record to a code file, optionally with line anchors or ranges, used as source-of-truth evidence for a concept.         | Duplicated prose, screenshot of code                       |
| **Materially Used Code Reference** | A Code Reference an agent explicitly marks as having influenced a decision, implementation direction, or docs-drift finding/fix, resolved to a commit-backed target in lineage. | Incidental file view, raw search result                    |
| **Documentation Directory** | The hybrid managed/editable top-level Workspace Document that acts as a table of contents for reusable documentation, linking to Level 2 entry point Documents with short descriptions and keyword lists. | Folder root, fully generated sitemap, search index         |
| **Documentation Methodology** | The three-level Workspace documentation structure: Level 1 Documentation Directory, Level 2 entry point Documents, and Level 3 focused deep-dive Documents.                | Wiki sprawl, arbitrary docs structure                      |
| **Docs Drift Automation** | A platform-defined periodic or manually scoped Automation family that spot-checks Documentation Methodology quality, fixes documentation issues it finds on a dedicated docs-drift branch, and moves the resulting PR through the standard review flow. | Per-change docs updater, implementation agent substitute   |
| **Daily Scoped Docs Drift Automation** | The default daily docs-drift Automation that inspects changed files since the last drift run, open docs-drift PRs, existing Code References, and docs-linked areas from the Documentation Directory. | Full repo scan, per-change updater                         |
| **Weekly Deep Docs Drift Automation** | The default weekly docs-drift Automation that uses a Director-style sectioning run to create a Plan with one docs-drift Task per section, then lets normal task dispatch and plan aggregation handle execution. | Daily drift run, single-agent whole-repo scan              |
| **Docs Drift Section Map** | A stable human-editable Markdown Document in the Operational Docs Library, maintained by the weekly deep Docs Drift Director, that maps repository and documentation topology into reusable docs-drift sections. | One-off split, hidden scheduler state                      |
| **Entry Point Document**  | A Level 2 Document that gives high-level knowledge across a topic area and links to focused Level 3 deep-dive Documents.                                                     | Overview page without links, category folder               |
| **Deep-Dive Document**    | A Level 3 Document that covers a specific concept in depth and stays focused enough to avoid irrelevant context for agents or humans.                                         | Catch-all page, transcript, hidden memory                  |
| **Task Progress Record**  | The task-owned durable progress state and first-class MVP model that stores bounded Checkpoints and Repair Context so later Sessions can resume with accurate context.      | Task-Local Continuity, document, prompt memory, transcript |
| **Task Progress Summary** | A bounded on-demand projection of the Task Progress Record that tells an agent what progress context exists and where to inspect more detail.                              | Hidden memory, transcript summary, generic blob            |
| **Checkpoint**            | A bounded task restart-context update that records todo or Acceptance Criteria status changes and may include a short note, key files, and key insights.                     | Heartbeat, log update, session status, savepoint, memory   |
| **Repair Context**        | Additional task-owned progress context inside the Task Progress Record, attached after a Repair Trigger as stable, provenance-rich items that steer the next repair assignment and remain distinct from Checkpoints. | Follow-Up Context, comment, failure text                   |
| **Follow-Up Context**     | Additional context used to create and steer a Follow-Up Task from prior terminal work that needs additional action; not part of the source Task Progress Record.           | Reopen context, repair context, bug report                 |
| **Follow-Up Task**        | A new task created from prior terminal work that needs additional action and linked to its source task.                                                                    | Reopened task, continuation task                           |
| **Follow-Up Source**      | The first-class provenance link from a Follow-Up Task to the prior terminal source Task, explicit source outcome, and, when relevant, source MergeRequest.                 | Prose reference, follow-up context                         |

## Execution and capacity

| Term                                 | Definition                                                                                                                                                                                                                         | Aliases to avoid                                      |
| ------------------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------- |
| **Host**                             | A customer-managed capacity source that connects outbound to Stoneforge and exposes runtimes.                                                                                                                                      | Machine, runner, server                               |
| **Host Agent**                       | The customer-managed service that maintains outbound connectivity, advertises capacity, and launches or controls execution on a Host.                                                                                              | Runner daemon, SSH agent                              |
| **Host Enrollment Token**            | A scoped secret used by a customer-managed Host Agent to bind its first outbound connection to a registered Workspace Host.                                                                                                        | Runner token, host credential                         |
| **Managed Sandbox**                  | A provider-created execution environment for per-assignment agent work.                                                                                                                                                            | VM, container, Daytona                                |
| **Runtime**                          | A reusable execution environment contract combining an execution location and execution mode; first-slice customer-managed Runtimes are statically bound to one Host.                                                             | Environment, runner, sandbox                          |
| **Execution Mode**                   | The way a runtime executes work, such as local worktree, container, sandbox, or VM.                                                                                                                                                | Runtime type, backend                                 |
| **Agent**                            | A stateless dispatchable AI capability that binds a harness and model to an acceptable Runtime set with numeric priority scores and optional throttle metadata.                                                                     | Bot, role, session                                    |
| **Director Agent**                   | An Agent acting under a director-category RoleDefinition to clarify intent, decompose work, shape plans, or supervise execution.                                                                                                   | Director role, planning bot                           |
| **Worker Agent**                     | An Agent acting under a worker-category RoleDefinition to implement, repair, or otherwise execute task work.                                                                                                                       | Worker role, coding bot                               |
| **Review Agent**                     | An Agent acting under a reviewer-category RoleDefinition to evaluate MergeRequests, produce Review Outcomes, or perform merge evaluation work.                                                                                     | Reviewer role, merge evaluator                        |
| **RoleDefinition**                   | The concrete prompt, tools, skills, hooks, category, and behavioral contract selected for a session.                                                                                                                               | Role, prompt, agent type                              |
| **RoleDefinition Version**           | The immutable RoleDefinition revision snapshot or version reference used by an Assignment so later RoleDefinition edits do not rewrite historical execution.                                                                       | Prompt history, role rollout                          |
| **Role Category**                    | A required controlled classification for a RoleDefinition, currently director, worker, reviewer, or custom.                                                                                                                        | Role tag, agent type                                  |
| **Director Role Category**           | The Role Category for RoleDefinitions intended for Director Agent work.                                                                                                                                                            | Director Agent type, planning tag                     |
| **Worker Role Category**             | The Role Category for RoleDefinitions intended for Worker Agent work.                                                                                                                                                              | Worker Agent type, coding tag                         |
| **Reviewer Role Category**           | The Role Category for RoleDefinitions intended for Review Agent work.                                                                                                                                                              | Review Agent type, merge evaluator category           |
| **Docs Reviewer RoleDefinition**     | A platform-created default Reviewer RoleDefinition for docs-only and docs-drift PRs, using the normal Review Outcome model with docs-focused review instructions.                                                                  | Separate review system, docs approver                 |
| **Custom Role Category**             | The Role Category for RoleDefinitions that do not fit director, worker, or reviewer categories; configuration-only until a product workflow needs a named actor phrase.                                                            | Custom Agent, automation actor                        |
| **Default RoleDefinition**           | The concrete RoleDefinition an Automation selects when no task-specific or one-off override is supplied.                                                                                                                           | Default role, automation role                         |
| **Role Override**                    | A task-specific or one-off replacement for an Automation's Default RoleDefinition.                                                                                                                                                 | Role tag, role preference                             |
| **Agent Command Surface**            | The Stoneforge-controlled API, CLI, and tool surface available to agents during assigned work for policy-checked control-plane/data actions and secret-backed capabilities.                                                        | Raw shell authority, unrestricted admin API, general agent tool use |
| **Session Command Credential**       | A short-lived, scoped Stoneforge credential minted for one Assignment/Session so an agent can call the Agent Command Surface without receiving a human credential or provider secret.                                               | Human token, provider token, ambient shell credential |
| **Tag**                              | A short label used for hard constraints on runtimes, agents, tasks, or automations, and for organization or role selection on RoleDefinitions.                                                                                      | Label, preference, score                              |
| **Unscoped Capability Tag**          | An additive hard tag requirement without a colon-delimited scope, such as `high-context` or `fast-review`.                                                                                                                          | Generic tag, additive label                           |
| **Scoped Routing Tag**               | A hard tag requirement using `scope:value` syntax, where the text before the first colon is the routing scope and the text after it is the value.                                                                                   | Scoped label, exclusive tag                           |
| **Agent-id Tag**                     | A stable read-only system-managed Agent Tag attached to one Agent so Required Agent Tags can pin work without a separate direct Agent constraint.                                                                                   | Specific agent constraint, direct agent selector      |
| **Required Agent Tag**               | A hard Agent Tag requirement carried by a Task, Automation, or one-off Dispatch Intent to constrain eligible Agents for provider, model, pool, capability routing, or Agent-id-style pinning.                                      | Role tag, agent preference, preferred agent tag       |
| **Agent Automation**                 | An Automation action that creates scheduler-evaluated Dispatch Intent for one concrete RoleDefinition.                                                                                                                             | Workflow evaluation, direct launch                    |
| **Code-First Automation**            | An Automation action that runs explicit workflow code instead of an agent session, either as platform-defined Stoneforge code or as a signed outbound call to user-hosted code.                                                     | Workflow evaluation, script job                       |
| **Platform Code-First Automation**   | A built-in Code-First Automation whose handler runs inside Stoneforge for core workflow behavior such as Merge Evaluation or failure Escalation.                                                                                    | Workflow evaluation action                            |
| **User-Defined Code-First Automation** | A Code-First Automation whose handler is user-hosted outside Stoneforge and invoked through signed outbound webhooks with idempotency.                                                                                              | Plugin, hosted Stoneforge workflow code               |
| **Dispatch Intent**                  | A durable request for the scheduler to evaluate a workflow action such as implement, review, merge evaluation, or escalation.                                                                                                      | Queue item, job request                               |
| **Qualified Dispatch Intent Phrase** | A descriptive phrase such as review Dispatch Intent, repair Dispatch Intent, merge-evaluation Dispatch Intent, or escalation Dispatch Intent used to clarify requested action without creating a separate canonical intent object. | Review Intent, Merge Evaluation Intent, Repair Intent |
| **Scheduler**                        | The durable internal system that evaluates readiness, queues work, leases capacity, places execution, retries, resumes, escalates, and propagates cancellation.                                                                    | Daemon, automation, runner                            |
| **Lease**                            | A scheduler-owned, time-bound reservation of agent throttle, runtime, and host or provider capacity for one Assignment lifecycle.                                                                                                  | Lock, assignment, slot                                |
| **Assignment**                       | The durable dispatch envelope for one implementation, repair, review, or merge-evaluation job.                                                                                                                                     | Run, task execution, session                          |
| **Qualified Assignment Phrase**      | A descriptive phrase such as implementation Assignment, repair Assignment, review Assignment, or merge-evaluation Assignment used to clarify workflow purpose without creating a separate canonical object.                        | Assignment subtype, job type object                   |
| **Session**                          | The concrete provider execution context under an Assignment; it may be a durable provider-side conversation/session record rather than a live connection.                                                                           | Assignment, run, agent, work session                  |
| **Session Connectivity**             | Adapter-declared Session behavior: `connectionless` Sessions continue by provider Session ID and prompt, while `connectionful` Sessions require an active provider or app-server connection for observation or control.             | Connection mode, session mode                         |
| **Adapter**                          | A provider-specific boundary that launches, resumes, cancels, observes, and reports execution facts without owning planning or policy decisions.                                                                                   | Integration, driver                                   |
| **Concurrency Limit**                | The maximum number of active Assignment leases or Sessions allowed through an Agent as a cost, token-budget, rollout, or A/B-test throttle.                                                                                        | Capacity, rate limit                                  |
| **Cost-Control Configuration**       | First-slice configuration that helps operators control spend indirectly, such as Agent concurrency limits, model/provider selection, Runtime selection, and active Assignment/Session counts.                                     | Billing, cost analytics, chargeback                   |

## Review and delivery

| Term                        | Definition                                                                                                                                                                                                                                                                                     | Aliases to avoid                                   |
| --------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------- |
| **MergeRequest**            | The provider-neutral internal artifact for review and merge flow.                                                                                                                                                                                                                              | PR, pull request, branch                           |
| **PR**                      | The GitHub-facing and first-slice user-facing representation of a MergeRequest.                                                                                                                                                                                                                | MergeRequest in internal model                     |
| **Merge Topology**          | The policy-selected branch and MergeRequest shape used to integrate Task and Plan work.                                                                                                                                                                                                        | Branch strategy, PR mode                           |
| **Task Branch**             | The source branch for work produced by a code-changing task.                                                                                                                                                                                                                                   | Feature branch, worker branch                      |
| **Plan Branch**             | An optional integration branch used when the Merge Topology aggregates task work at the plan level.                                                                                                                                                                                            | Staging branch, aggregate branch                   |
| **Workspace Target Branch** | The branch configured as the merge target for completed workspace work.                                                                                                                                                                                                                        | Main, default branch                               |
| **Staging Branch**          | A future first-class branch concept for requiring task and plan work to merge into a staging integration branch before policy-gated promotion to the Workspace Target Branch. In the current MVP, teams can approximate this by configuring the staging branch as the Workspace Target Branch. | Parent branch, plan branch                         |
| **Unplanned Task Flow**     | The Merge Topology where a task branch opens a PR directly to the Workspace Target Branch.                                                                                                                                                                                                     | Direct PR flow                                     |
| **Planned Task Flow**       | A Merge Topology where planned task branches either merge directly to the Workspace Target Branch or aggregate through an optional Plan Branch before merging onward.                                                                                                                          | Aggregation flow                                   |
| **Verification Run**        | A Stoneforge-owned aggregate verification record for one MergeRequest head SHA, derived from one or more Provider Checks and used as a merge-policy input.                                                                                                                                     | CIRun, CI run, build, check run, status            |
| **Provider Check**          | A raw external verification signal observed from a provider, such as a GitHub check/status or a Jenkins/Buildkite result; Stoneforge policy determines whether it is required for the Verification Run.                                                                                        | Verification Run, Stoneforge Policy Check          |
| **Provider Check Log**      | Full provider-hosted CI log content for a Provider Check, fetched only when available through documented GitHub APIs rather than through provider-specific integrations or eager storage.                                                                                                     | Verification Run, Provider Check metadata          |
| **Provider Check Failure Context** | Bounded failure context automatically stored for a failed required Provider Check, including check name, conclusion, timestamps, provider URL, job/step names when available, and a failure excerpt or summary when available.                                                       | Full CI log, raw Actions archive                   |
| **Local Actions Verification** | Agent-run GitHub Actions verification through the `act` CLI inside a capable Runtime before pushing repair commits, using the same low-risk preview/dev secret boundary as previews. | GitHub rerun, workflow dispatch, hosted CI         |
| **Mergeability**            | The provider/source-control condition that a MergeRequest can be merged at this moment, including conflicts, stale branch state, branch protection, locks, and required provider permissions.                                                                                                  | Verification Run, policy check, approval           |
| **Branch Health**           | The broader condition of a task, plan, or integration branch over time, including drift, risky divergence, repeated rebase failures, branch deletion, or stale-base risk that may require repair before immediate Mergeability is evaluated.                                                   | Mergeability, CI, branch status                    |
| **Branch Update**           | The action of bringing a task, plan, or integration branch up to date with its configured target branch through merge, rebase, or provider-supported update.                                                                                                                                   | Repair, merge attempt, sync                        |
| **Unauthorized Branch Mutation** | A detected agent or service-actor change to a GitHub branch outside the Assignment's scoped branch/worktree or outside policy-allowed provider operations.                                                                                                                              | Normal push, merge conflict, manual edit           |
| **Editor Commit**           | A human-authored repository branch commit made through a Stoneforge code or Document editing surface, optionally linked to a Task, Plan, MergeRequest, Document, or docs PR context.                                                                                                      | Agent commit, merge action, task completion        |
| **Review Outcome**          | The recorded result of Human Reviewer or Review Agent evaluation, either Review Approved or a Change Request.                                                                                                                                                                                  | Review comment, PR state                           |
| **Review Approved**         | A Review Outcome meaning the reviewer found the work acceptable; this is the approval event in Stoneforge.                                                                                                                                                                                     | Separate approval record                           |
| **Stale Review**            | A Review Approved outcome that remains in history but no longer satisfies Required Review policy because GitHub-compatible review freshness rules require newer review.                                                                                                                        | Deleted approval, failed review                    |
| **Provider Review**         | The provider-side PR review representation of a Stoneforge Review Outcome, such as a GitHub approve or request-changes review posted from a Review Agent outcome.                                                                                                                              | Stoneforge Review Outcome, policy check            |
| **Repair Trigger**          | Any review, verification, mergeability, policy, or branch-health condition that requires task repair before completion.                                                                                                                                                                        | Failure, rejection, reopen                         |
| **Change Request**          | A repair trigger produced by a Human Reviewer or Review Agent.                                                                                                                                                                                                                                 | Rejection, failure                                 |
| **Merge Blocker**           | A review, verification, mergeability, branch-health, Required Review, or policy condition that prevents a MergeRequest from becoming merge-ready until resolved.                                                                                                                               | Gate failure, broken check, merge failure          |
| **Required Review**         | A policy requirement that one or more qualifying Review Approved outcomes exist before merge or another sensitive action, aligned with GitHub-style required PR review rules.                                                                                                                  | Approval Gate, separate approval record            |
| **Required Human Review**   | A Required Review that must be satisfied by a Human Reviewer.                                                                                                                                                                                                                                  | Human Approval Gate, separate approval role        |
| **Required Agent Review**   | A Required Review that may be satisfied by an authorized Review Agent.                                                                                                                                                                                                                         | Agent Approval Gate, automated approval            |
| **Stoneforge Policy Check** | The provider-facing check or status Stoneforge publishes to communicate the current relevant Policy Decision for a MergeRequest head SHA.                                                                                                                                                      | CI check, approval check, policy decision          |
| **Merge Evaluation**        | The workflow action that refreshes Mergeability and runs Policy Evaluation for a MergeRequest.                                                                                                                                                                                                 | Merge check, final review                          |

## Automation and workflow triggers

| Term                            | Definition                                                                                                                                                                                                                      | Aliases to avoid                                               |
| ------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------- |
| **Automation**                  | A durable user-facing workflow trigger that creates controlled Stoneforge workflow intent.                                                                                                                                      | Scheduler, bot, daemon                                         |
| **Automation Version**          | An immutable version or effective snapshot of an Automation definition used by AutomationRuns so historical runs and manual replays remain explainable after edits.                                                             | Current automation config, mutable trigger                     |
| **AutomationRun**               | A durable record of one automation trigger evaluation and action attempt, including trigger source, policy evaluation, idempotency key, target object, action type, status, attempts, result, and linked follow-up objects.    | Assignment, Session, hidden job log                            |

## Relationships

- An **Org** owns one or more **Workspaces**; a **Workspace** belongs to exactly one **Org**.
- First-slice **Org** administration is minimal: create or select Org, manage Workspaces, basic membership and roles, and GitHub org/repo permission projection. Advanced org billing, SSO enforcement, audit export, cross-Workspace policy inheritance, and org-wide analytics are out of scope.
- First-slice setup uses a guided onboarding wizard based on the `reference/smithy-next` prototype onboarding flow. It should create or select the first **Org**, create the **Workspace**, connect the GitHub **Repository**, bootstrap default **RoleDefinitions**, configure the first **Runtime** path and **Agent**, select policy, and surface readiness blockers. Additions are allowed for missing first-slice requirements, but existing prototype setup steps should not be removed.
- A seeded local demo or sandbox **Workspace** may exist for local OSS/dev mode when cheap, but core MVP acceptance requires real GitHub-backed onboarding through the onboarding wizard.
- First-slice local OSS/dev uses SQLite or local Postgres, a clearly marked single-user principal, local filesystem/dev secret storage, and real GitHub App configuration for repository testing. Fake providers are not part of first-slice product acceptance.
- First-slice self-hosted team mode uses Postgres, durable object or file storage for logs and artifacts as needed, a configured GitHub App, OAuth or OIDC human authentication, secret storage, worker and Host connectivity, and readiness checks. Kubernetes, high availability, backup/restore automation, managed upgrades, billing, multitenant SaaS operations, and a Stoneforge-hosted cloud service are out of scope.
- V1 and Smithy data migration is out of scope for the first slice. `reference/v1` is historical behavior/code context, `reference/smithy-next` is UI/workflow direction, and first-slice setup creates clean V2 objects rather than importing legacy production data.
- Broad Workspace export/import, audit export, portable project archives, backup/restore automation, and migration tooling are out of scope as product requirements. Minimal JSON or debug exports may exist as engineering conveniences only.
- A **Workspace** maps to exactly one **Repository** in the first slice.
- Cross-repository Workspaces are out of scope for the first slice and should be deferred until cross-repo permissions, branch/PR coordination, code references, and audit scope are designed.
- A **Workspace** owns its **Tasks**, **Plans**, **Documents**, **Hosts**, **Runtimes**, **Agents**, **RoleDefinitions**, **Automations**, **Assignments**, **Sessions**, **MergeRequests**, **Verification Runs**, **Policy**, and **AuditEvents**.
- A **Host** is registered to one **Workspace** and may reconnect, be disabled, or be removed in the first slice.
- Disabling a **Host** stops new dispatch and leases to that Host but does not implicitly stop running **Assignments** while the Host remains connected.
- Removing or force-disconnecting a **Host** with running **Assignments** requires explicit confirmation and an explicit stop/interruption outcome.
- Unexpected **Host** disconnect creates **Contact Loss** for affected connectionful **Sessions** until timeout or reconnect reconciliation proves whether execution is still healthy or must resume/re-dispatch.
- During **Contact Loss**, Stoneforge must not send new prompts or commands for the owning **Assignment**.
- During reconnect reconciliation, the **Host Agent** and **Adapter** report observed facts, while the Stoneforge control plane records the authoritative **Session** and **Assignment** state transitions.
- A **Task** belongs to zero or one **Plan**; a **Plan** contains many **Tasks**.
- A **Task** may depend on many other **Tasks**; unresolved **Dependencies** block **Readiness**.
- A **GitHub Issue Intent** may remain intake-only, become an **Issue-Backed Draft Task**, or be promoted to a dispatchable **Task** according to **Workspace** policy and Director or human triage.
- GitHub Issue sync may be bidirectional without implying a strict one-Issue-to-one-Task mapping; one Issue may decompose into multiple Tasks, and one Task may link to a primary Issue plus secondary Issue links.
- Minimum first-slice GitHub Issue sync includes title, body, author, labels, assignees, milestone, state, URL/number, timestamps, and comments needed for Director triage. GitHub PR review comments and top-level PR comments should be visible enough in Stoneforge to support review and repair context. Stoneforge may write linked-task or status comments back to GitHub Issues; field-perfect sync, GitHub Projects sync, issue forms/schema mapping, full bidirectional comment sync, threaded resolution parity, perfect comment editing/deletion sync, and complex conflict resolution are out of scope.
- PRs created from linked **Tasks** should reference the primary linked GitHub Issue and relevant secondary Issues. Task PRs may use issue-closing keywords only when the linked Issue has exactly one linked Task; when one Issue decomposes into multiple Tasks, task PRs use non-closing references and the plan PR links back to the source Issue for plan-level closure.
- **Stoneforge Issue Labels** may mirror status, priority, and complexity to GitHub Issues when policy enables label writeback; those labels are Stoneforge-owned projections, not GitHub-side commands. Direct GitHub edits are reconciled back or surfaced as quiet conflicts, and Stoneforge remains the workflow source of truth.
- `backlog` is UI/product grouping language for non-dispatchable **Tasks**, not a separate canonical Task state.
- **Issue Intake Policy** defaults depend on policy preset: autopilot imports Issues to Director triage, while supervised imports Issues as non-dispatchable draft Tasks that may appear in backlog/intake UI.
- **Issue-Backed Draft Tasks** and other non-dispatchable imported Tasks cannot become ready for dispatch without a human status change or human instruction to Director triage.
- **External Issues** default to non-dispatchable draft state with a visible external-origin indicator included in backlog/intake UI and Director context.
- **Trusted Maintainer Labels** `stoneforge:auto-dispatch` and `stoneforge:auto-triage` may override normal intake and triage-approval policy unless the Workspace disables trusted-maintainer label overrides.
- `stoneforge:*` labels may exist on GitHub Issues as provider facts, but Stoneforge ignores their intake effects when they are applied by unlinked users, external contributors, or actors without sufficient GitHub-backed repository permission.
- Ignored `stoneforge:*` labels should produce quiet Stoneforge UI warnings and workflow/audit lineage, not public GitHub comments by default.
- A **Plan** must be **active** before its **Tasks** may become dispatchable.
- A **Document** may be referenced by many **Tasks**, **Plans**, **Assignments**, **MergeRequests**, or **Automations**.
- **Documents** are reusable Workspace context, not task progress records, transcripts, or hidden prompt memory.
- First-slice **Documents** are grouped into at least two **Document Libraries**: the **Workspace Documentation Library** rooted at the Documentation Directory, and the **Operational Docs Library** for system-managed operational Documents.
- The Documents UI defaults to the **Workspace Documentation Library** and exposes the **Operational Docs Library** through a clear library selector or filter such as "Operational/System." Operational docs are visible but separated, not hidden behind a deep admin-only path.
- **Operational Docs Library** Documents default to Stoneforge-only source mode in the first slice. They are Stoneforge operational metadata, not team docs meant to live in the repository docs root.
- Document visibility and access inherit Workspace policy in the first slice. Document-specific ACLs are out of scope; sensitive operational knowledge that requires different access belongs in a separate Workspace or future document-level controls.
- First-slice **Documents** use Markdown/plain text as the primary editable type and support URL/reference documents for external resources.
- **URL/Reference Documents** may participate in the Documentation Directory and Level 2/Level 3 Documentation Methodology like other Documents, but they store external-resource metadata plus optional summaries or notes. Stoneforge does not automatically crawl or ingest arbitrary external pages in the first slice.
- Agents may open URL/Reference Document targets only when their RoleDefinition and tool access allow it. Durable knowledge learned from an external source should be summarized into a normal Document when it matters long term.
- External knowledge becomes a normal Markdown **Document** when the team will rely on it without always revisiting the external source; that Document should include summary, rationale, and source link. Use a **URL/Reference Document** when the external page itself remains the thing to consult and Stoneforge only needs a pointer plus notes.
- **Document Source Mode** may be Stoneforge-only, repo-backed, or mirrored. The default for repository-linked Workspaces is mirrored docs, with the repository docs folder selected during repository onboarding.
- Repository onboarding should detect common documentation paths such as `docs/`, `documentation/`, `.github/`, and root Markdown docs, then ask the user to confirm or choose one **Primary Docs Root**. If no docs folder exists, onboarding should offer to create `docs/` with the Documentation Directory. Multiple docs roots are out of scope for the first slice.
- Stoneforge **Documents** are the UI and agent context surface. Repository docs files are source-controlled artifacts. For **Repo-Backed Documents** or **Mirrored Documents**, docs-drift PRs change repository files and Stoneforge updates or reindexes corresponding Document objects from the merged source.
- **Document Version History** is source-mode aware. Stoneforge-only Documents use Stoneforge-owned content history. Repo-backed and mirrored Documents derive content history from Git commits and PRs, while Stoneforge records indexing events, metadata changes, category/tag changes, link-resolution changes, and agent-context selection events.
- For **Mirrored Documents**, repository PR merge is the authoritative write path. Stoneforge UI edits create a new docs PR or add commits to an existing docs PR, and merged repository state is imported back into the Document object.
- The Documents UI may show or hide pending PR changes, or let users select a branch or docs PR context for viewing and editing pending Document changes. Conflicts are resolved through normal source-control merge conflict handling.
- **Document Links** use one canonical methodology across all source modes: normal relative Markdown file links. In mirrored or repo-backed modes they resolve to files under the Primary Docs Root; in Stoneforge-only mode they resolve through the Document's workspace-relative path or filename. The Documents UI should make them clickable and may offer autocomplete or link insertion, but users should not need to hand-author Stoneforge IDs or slugs.
- **Document Categories** are a small optional system set for UI and search facets, with one primary category per Document. **Document Tags** are freeform, Workspace-scoped, and many-to-one. Nested document category trees are out of scope for the first slice.
- **Document Search** searches title and body, supports filters for category, tags, content type, linked workflow object, creator/editor, and updated time, and uses BM25 ranking in both SQLite and Postgres. Postgres deployments use the `pg_textsearch` extension for this behavior.
- Every Workspace should maintain one **Documentation Directory** as Level 1 of the **Documentation Methodology**. It links to **Entry Point Documents** with short descriptions and keyword lists for browse-first discovery.
- The **Documentation Directory** is hybrid managed/editable: users and agents may edit it like a normal Document, while Stoneforge treats it as the managed Workspace entry point and provides link validation, missing Level 2 entry point warnings, stale keyword or description warnings, and special Docs Drift Automation attention. It should not be fully generated and overwritten.
- **Entry Point Documents** are Level 2 overview documents that link to Level 3 **Deep-Dive Documents**. **Deep-Dive Documents** are focused concept documents. Important documentation should be reachable within three steps without forcing broad context into one large Document.
- **Code References** act as a Layer 4 reference target in the Documentation Methodology. Any Document level may link directly to code files or line ranges when code is clearer or more efficient than duplicating the concept in prose. Code remains the ultimate source of truth; Documents make context-building more effective and explain why decisions were made.
- **Code References** in Document content use repository-relative file paths with optional line anchors or ranges. Stoneforge lineage resolves **Materially Used Code References** to the relevant commit SHA when recorded, while the UI displays the friendly path and line range.
- **Code References** receive lightweight first-slice validation: referenced file existence on the relevant branch and warnings for missing paths or obviously stale line anchors/ranges. Semantic validation that code still proves the documented claim is left to review and Docs Drift Automation.
- The lightweight editor may commit human changes on any Workspace branch when provider permissions and branch protections allow it. Contextual Task, Plan, MergeRequest, or Document views may preselect relevant branches but do not restrict editing to those branches.
- Unlinked **Editor Commits** are allowed as repository branch edits and do not automatically create a Task or review workflow. If the commit affects an observed MergeRequest or workflow branch, normal provider observation, Verification Run, Required Review freshness, Mergeability, and policy re-evaluation apply. Direct target-branch commits are governed by GitHub permissions and branch protection rather than a synthetic Stoneforge merge workflow.
- Stoneforge may offer agent-assisted commit message generation in the editor; generated messages are suggestions reviewed and submitted by the human.
- First-slice **Code References** are limited to the Workspace primary repository. Cross-repository code references should use URL/Reference Documents or ordinary external links until cross-repo validation, commit resolution, permissions, and review/audit scope are designed.
- Agents receive Documents through explicit context selection first, including linked Documents, human-attached Documents, and Director-selected Documents. Agents may browse through the Documentation Directory or use Document Search through the Agent Command Surface, but broad Workspace document context is not injected into every Session by default.
- **Document Browsing** and **Document Search** are distinct behaviors. Browsing is open-ended full-Document reading and relative-link traversal, usually starting from the Documentation Directory. Search is targeted lookup for a specific topic, concept, keyword, or filtered set of Documents.
- Stoneforge automatically records Document read, search, and browse events for observability. Agents should explicitly mark a **Materially Used Document** immediately after relying on it for a decision or implementation direction. This is decision and implementation lineage, not a documentation-update marker.
- **Materially Used Document** markers require a short reason, visible to Review Agents and humans inspecting the Task or MergeRequest. The reason should explain why the Document mattered without becoming a report.
- **Materially Used Document** markers are independent from documentation Acceptance Criteria. Documentation Acceptance Criteria ask whether task changes require Document updates, new Documents, or a no-docs-needed rationale.
- Agents may mark **Materially Used Code References** when code files or line ranges influenced a decision, implementation direction, or docs-drift finding/fix. This is especially important for Docs Drift Automation, where code is usually the evidence for code-to-docs drift.
- Documents materially used by agents through browsing or search should be cited or linked in Execution Lineage.
- Agent use of Documents is mandatory in bounded workflows: Directors consult the Documentation Directory for non-trivial or concept-changing decomposition, Workers consult linked or selected Documents and the Documentation Directory when documentation Acceptance Criteria apply or documented concepts are touched, and Review Agents check relevant Documents when validating documentation Acceptance Criteria or concept changes.
- Agents are not required to read every linked Document when clearly irrelevant, but they should record which Documents materially informed the work in lineage.
- Documentation drift prevention is primarily task-local: Tasks should include **Acceptance Criteria** requiring documentation updates when task work changes durable code or product concepts, the task-completing agent should update relevant Documents while implementation context is fresh, and Review Agents should verify those Acceptance Criteria during review.
- Every code-changing or concept-changing **Task** should include a default documentation **Acceptance Criteria**: update relevant Documents or record why no documentation change is needed. Pure investigation, review, or administrative Tasks may omit it when they do not affect durable code or product concepts.
- A Worker satisfies documentation Acceptance Criteria by updating relevant Documents or recording a short no-docs-needed rationale. The Review Agent verifies that decision during review.
- The **Docs Drift Automation** family is not a per-code-change docs updater. The **Daily Scoped Docs Drift Automation** runs every configurable X hours, defaulting to every 24 hours overnight, and inspects changed files since the last drift run, open docs-drift PRs, existing Code References, and docs-linked areas from the Documentation Directory. It can also be manually triggered from Document create/edit flows scoped to the specific change.
- The **Weekly Deep Docs Drift Automation** periodically assesses broader codebase/documentation alignment. It may split the codebase into sections and dispatch multiple agents, each responsible for one section, to avoid context overload and quality decay from a single whole-codebase assessment.
- The weekly deep flow starts with a Director-style sectioning run that inspects repository structure and Documentation Directory topology, creates a section split, and creates a Plan with one docs-drift Task per section. Normal Scheduler dispatch handles those Tasks; task PRs aggregate into a single plan branch and plan PR by default.
- Weekly deep docs-drift Plans follow the same Workspace Plan activation policy as all other Plans. There is no docs-specific human activation policy; either all Plans in the Workspace require human activation under effective policy, or this Plan does not require special activation.
- The weekly deep Docs Drift Director creates and maintains a stable **Docs Drift Section Map** Document in the **Operational Docs Library**. It reuses the stable section map when possible and revises it when repository structure or Documentation Directory topology changes. A separate section-boundary change rationale is not required because normal agent Session lineage captures Director reasoning.
- The **Docs Drift Section Map** is authorized human-editable Markdown without schema validation in the first slice. Human edits are versioned in normal Document lineage. The next Director sectioning run interprets the map and may preserve, repair, or revise it when it is unclear, stale, or inconsistent with repository or documentation topology.
- Agents running **Docs Drift Automation** should fix documentation issues they find while they have the relevant context, accumulating changes on a dedicated docs-drift branch and draft PR. When the Automation completes, the PR is marked ready and follows the standard MergeRequest review, verification, Required Review, and merge flow according to Workspace policy.
- Docs Drift agents record **Materially Used Code References** when code evidence drives a finding or fix. **Materially Used Document** markers are still used when existing Documents influenced the fix, but they are not the primary evidence path for code-to-docs drift.
- Docs-drift Tasks are docs-only in the first slice. Code issues discovered during docs-drift work are reported to a Director for triage or follow-up Task creation, not patched by the docs-drift Worker.
- Docs-drift Tasks may create new Documents when the Documentation Methodology calls for it. Section Workers may create missing Level 2 or Level 3 Documents, update the Documentation Directory, add Code References, and update relevant links. In mirrored mode, new or edited Documents flow through the docs PR and standard review process.
- Daily scoped Docs Drift Automation uses one docs-drift PR per run. Weekly deep docs drift aggregates section task PRs into one plan branch and one plan PR by default. Risky or unrelated section changes may be split into follow-up Plans or Tasks by the Director or policy.
- Docs-drift PRs use the same **MergeRequest** framework as code PRs, while policy may distinguish docs-only changes. Defaults require Review Agent review for docs-only PRs; Human Review follows the Workspace policy preset and any docs-only exemptions.
- Stoneforge creates a default **Docs Reviewer RoleDefinition** for docs-only and docs-drift PRs. It remains in the reviewer role category and uses the normal **Review Outcome** model; its prompt/checklist focuses on clarity, correctness, link and Code Reference quality, Documentation Methodology fit, and whether doc deletions are justified. Workspaces may edit or delete it like other default RoleDefinitions.
- Code PRs that include documentation changes keep the normal code **Review Agent** as the sole automatic Review Agent. Stoneforge does not add a second docs-focused review automatically for mixed code/docs PRs; normal code Reviewer RoleDefinitions include docs-focused review pointers for documentation Acceptance Criteria, Document Link quality, Code Reference quality, and no-docs-needed rationales.
- Docs-only Human Review exemptions are explicit Workspace policy toggles. `supervised` defaults to Human Review for docs-only PRs unless the Workspace enables a docs-only exemption. `autopilot` defaults to agent-only review for docs-only PRs, subject to applicable Provider Checks, Mergeability, and policy checks.
- Docs-only classification is determined by changed file extensions, not docs-root path. Docs-only policy applies when every changed file is a recognized Markdown or text documentation file, such as `README.md`, `.md`, `.mdx`, `.txt`, or another configured text-doc extension. Docs-drift Tasks and PRs may still carry docs-only intent metadata, but the final classification uses extension-based changed-file verification.
- The docs-only extension set is Workspace-configurable and defaults to a small list such as `.md`, `.mdx`, `.txt`, and `.rst`.
- Config-like files such as `.json`, `.yaml`, and `.yml` are excluded from docs-only defaults because they may affect runtime behavior. Workspaces may deliberately add structured documentation extensions if needed.
- Docs-only classification treats creates, edits, and deletions the same way: if every changed file has a recognized docs-only extension, the PR remains docs-only. Risky docs deletions are handled by review policy and Review Agent output rather than reclassification as code.
- Changing the docs-only extension set warns when open MergeRequests would be reclassified and requires authorized confirmation. Open MergeRequests are re-evaluated under the new policy; terminal or merged MergeRequests keep historical classification.
- Stoneforge does not publish a dedicated docs-specific GitHub check/status for docs-only PRs in the first slice. Docs-only PRs use the normal **Stoneforge Policy Check** and observed GitHub/provider checks as **Provider Checks** inside **Verification Runs**. Docs quality, Document Link, and Code Reference findings appear in Stoneforge MergeRequest UI and Review Agent output rather than a separate GitHub check.
- **Docs Drift Automation** must account for open unmerged docs-drift PRs. Later runs treat proposed changes in those PRs as pending context, avoid repeating the same fixes, and either skip overlapping findings, append to the existing branch when safe, or create only non-overlapping follow-up work according to policy.
- Workspace policy may disable the **Documentation Methodology**. When disabled, default RoleDefinition documentation-update instructions are removed or hidden, the default documentation Acceptance Criteria is not added to Tasks, Docs Drift Automations are disabled, and manual docs-drift trigger suggestions on Document edits are disabled.
- **Task Progress Record** belongs on the **Task** and may reference the relevant **Assignment** and checkpoint event.
- A **Task Progress Summary** is an on-demand projection of the **Task Progress Record**, not a second source of truth; it should cite or point to the relevant Checkpoints, repair context, remaining work, and other structured details.
- The MVP **Task Progress Record** stores only **Checkpoints** and **Repair Context**; **Task Progress Summary** is generated outside the persisted record on demand.
- Same-Task resumed **Assignments** and repair **Assignments** automatically include a generated **Task Progress Summary** in prompt assembly.
- **Follow-Up Tasks** use **Follow-Up Context** rather than automatically inheriting the source Task's **Task Progress Summary**.
- First-slice **Checkpoint** and **Repair Context** redaction is optional unless it can reuse the transcript/log redaction path with little added complexity.
- Full provider transcripts are **Session** or **Execution Lineage** records, not **Task Progress Record** content; Task Progress Records may reference transcript/log/event records when needed.
- Raw provider transcripts, provider logs, CI logs, Automation payloads, preview logs, and runtime logs have Workspace-level retention policy with configurable retention and conservative defaults.
- Raw provider transcripts/logs receive best-effort automatic **Transcript Redaction** for known secret patterns before storage or display, and authorized users may manually redact transcript/log ranges after storage.
- Stoneforge does not retain an unredacted original transcript/log by default after automatic redaction; any future forensic raw-retention mode must be explicit, high-risk, self-hosted policy.
- Manual **Transcript Redaction** permanently replaces the stored/displayed range in normal Stoneforge flows; restoration requires external backup/restore outside the product flow.
- Redacted transcript/log ranges remain visible as placeholders in **Session** and **Execution Lineage** views, with authorized metadata such as actor, timestamp, and reason.
- An **Automation** creates **Dispatch Intent** or **Outbound Automation Webhook** calls; it does not directly create **Leases**, **Assignments**, or **Sessions**.
- An **AutomationRun** records why an **Automation** fired and what action it attempted; agent-based AutomationRuns may create **Dispatch Intent** that later produces **Assignments** and **Sessions**, but the AutomationRun is not itself an Assignment or Session.
- Each **AutomationRun** snapshots the effective **Automation** definition or references an immutable **Automation Version** at run creation. Later edits do not change historical run interpretation or exact manual replay behavior.
- Running, delivering, blocked, and retrying **AutomationRuns** keep their creation **Automation Version** or snapshot; edits affect future runs only.
- Disabling an **Automation** prevents new trigger-created **AutomationRuns** but does not stop existing runs by default; canceling active runs is explicit.
- Deleting an **Automation** uses archive or soft-delete semantics in the first slice; versions and historical **AutomationRuns** are preserved.
- Archived or soft-deleted **Automations** keep historical **AutomationRuns** inspectable, but direct manual re-run requires restoring the Automation or creating a new Automation from the historical version.
- **AutomationRun** uses a unified `succeeded` state for successful automation actions. Action-specific outputs, such as created Dispatch Intents or completed webhook delivery, live on the result and linked records.
- **AutomationRun** `blocked` may be resumed when policy, eligibility, idempotency, or configuration facts change and Workspace policy allows re-evaluation; it is not the same as `failed`.
- Long-running code-first **AutomationRuns** complete by signed callback only in the first slice; Stoneforge does not poll external status endpoints. Callback receipt returns 200 after authentication, idempotent acceptance, and durable recording.
- Completion callbacks finalize **AutomationRuns** only; user-defined **Code-First Automation** uses the Stoneforge API for workflow actions.
- An **AutomationRun Credential** authenticates user-defined code-first automation calls to the Stoneforge API. It is scoped to the Workspace, AutomationRun, configured target objects, allowed API action categories, and expiry, and records service-actor lineage for every call.
- **AutomationRun Credentials** are usable from outbound delivery through `running`, including final API calls before completion callback, and are revoked when the AutomationRun reaches a terminal state.
- User-defined **Code-First Automation** may create **Tasks** and **Plans** through the Stoneforge API when credential scope allows. Workspace policy or Automation configuration controls whether created work moves forward, requires Director triage, becomes draft/non-dispatchable, or becomes ready/active.
- For automation-created **Tasks** and **Plans**, Workspace policy is the ceiling and Automation configuration can only narrow it. Director triage is requested through an explicit Stoneforge API action that creates scheduler-evaluated Director **Dispatch Intent** when policy allows.
- Director triage requests from **Code-First Automation** may target objects created by the same **AutomationRun** or existing **Tasks**, **Plans**, and imported **Issues**, constrained by **AutomationRun Credential** scope.
- Repeated Director triage requests from **Code-First Automation** are deduped by target object, triage purpose or reason, source Automation/AutomationRun, and unresolved Director **Dispatch Intent** status.
- Manual re-run of an **AutomationRun** from any terminal state creates a new linked AutomationRun and never mutates the old run.
- Manual re-run replays the original trigger context by default; run-with-edited-inputs creates a new manual **AutomationRun** with the input diff recorded and requires Automation edit/create authorization.
- **AutomationRun** history should be searchable and filterable by Automation, target or linked object, trigger type, action type, state, source actor/provider, time window, and correlation or idempotency key.
- **AutomationRun** detail shows raw trigger, outbound request, and callback payloads in the first slice. Those payloads must not contain secrets and do not require special first-slice redaction or permissions.
- Automation payload validation may warn on obvious secret-looking field names such as `secret`, `token`, `password`, `apiKey`, or `privateKey`, but heavy DLP is out of scope for the first slice.
- First-slice user-defined **Automations** use **Curated Automation Triggers** such as Task state changed, Plan state changed, MergeRequest opened or updated, Review Outcome recorded, VerificationRun changed, AutomationRun changed, schedule, and inbound signed webhook. Arbitrary "any object changed" triggers are out of scope.
- First-slice **Inbound Webhook Triggers** use one unique endpoint per Automation. Shared Workspace webhook endpoints are out of scope unless added later.
- **Inbound Webhook Signing Secrets** may be per-webhook or workspace-wide. Workspace or Org policy may disable workspace-wide signing-secret use when stricter isolation is required.
- **Inbound Webhook Signing Secrets** support rotation with an overlap window and last-used metadata before old-secret retirement.
- **Outbound Webhook Signing Secrets** may be per-destination or workspace-wide. Workspace or Org policy may require per-destination secrets, and rotation uses an overlap window plus last-used metadata.
- **Outbound Automation Webhook** delivery treats `2xx` as success, `408`, `429`, and `5xx` as retryable, and other `4xx` responses as terminal failure. Retries use bounded exponential backoff.
- **Outbound Automation Webhook** handlers use a typed response contract for accepted, succeeded, or failed outcomes. The **Stoneforge SDK** is recommended, not required; the protocol remains documented and manually implementable.
- The outbound automation webhook protocol is language-agnostic. The first-slice **Stoneforge SDK** ships for TypeScript first, with other languages left for later or manual implementation.
- The first-slice **Stoneforge SDK** includes Automation handler support for signature verification, typed responses, callbacks, AutomationRun Credential API client setup, and thin helpers for common actions such as create Task, create Plan, request Director triage, link objects, complete or fail an AutomationRun, and read scoped context. It is not a workflow DSL.
- The TypeScript **Stoneforge SDK** is in first-slice scope, but broader SDK expansion beyond the required first-slice API and Automation helper surface is out of scope.
- First-slice product-level API contracts are required only where external actors depend on them: the **Agent Command Surface**, **Stoneforge SDK** and Automation handler APIs, GitHub webhook/provider integration boundaries, and minimal UI/backend private APIs needed by the app. A fully public REST or GraphQL API for every object is out of scope. Internal domain and persistence APIs remain implementation detail.
- Schedule-based **Automations** support simple intervals and cron expressions with explicit required timezone; schedule triggers record evaluated next-run time and timezone in AutomationRun/source metadata.
- Missed schedule runs after downtime are not backfilled in the first slice; Stoneforge records missed-run visibility, evaluates the next future run, and users can manually run the Automation if recovery is needed.
- Manual runs of schedule-based **Automations** create separate **AutomationRuns** and do not move or reset the next scheduled run time.
- **AutomationRun changed** triggers require explicit filters for state, action, and source; direct self-triggering is prevented by default; idempotency keys are required; and automation chain depth is capped per root correlation ID.
- Exceeding the automation chain-depth cap creates a blocked **AutomationRun** with reason `automation_chain_depth_exceeded`, root correlation ID and parent-run lineage, and UI visibility rather than silently dropping the event.
- User-defined automation chain-depth cap is **Workspace** policy with a conservative default of `3`; **Platform Automations** may use separate internal safeguards.
- A **Platform Automation** is the built-in rule that requests a core workflow action; the **Scheduler** owns readiness checks, capacity leasing, Assignment creation, and Session launch.
- **Code-First Automation** names the user-hosted-code model; **Outbound Automation Webhook** is the signed transport boundary for that model.
- An **Idempotency Key** may apply to inbound triggers, outbound automation webhooks, or future external action deliveries; it is not webhook-only language.
- A **Dispatch Intent** may request implementation, review, repair, merge evaluation, or escalation through its action/type and target; those are not separate canonical intent objects unless their ownership, lifecycle, or scheduler behavior diverge later.
- The **Scheduler** resolves exactly one **RoleDefinition**, one **Agent**, and one **Runtime** before execution starts.
- A **RoleDefinition** has exactly one **Role Category**; **Tags** may organize or select RoleDefinitions but must not constrain Agent or Runtime placement.
- Any **RoleDefinition** may be combined with any **Agent** for an **Assignment**.
- RoleDefinitions are editable Workspace objects with simple version/history tracking. Authorized users may create, edit, disable, or delete them. Assignments snapshot or reference the **RoleDefinition Version** they start with.
- Advanced marketplace roles, role-template libraries, multi-version rollout controls, role A/B testing, and cross-workspace role sharing are out of scope for the first slice.
- Provider/model routing such as Codex vs Claude belongs in **Agent Tags** and **Required Agent Tags**, not in RoleDefinitions.
- **Required Agent Tags** are hard eligibility constraints only in the first slice; preferred Agent tags and tag scoring are out of scope.
- Tags containing a colon are **Scoped Routing Tags**; tags without a colon are **Unscoped Capability Tags**.
- First-slice first-class **Scoped Routing Tag** dimensions are `provider`, `model-family`, `model`, and `agent`.
- `provider` selects provider/harness family, `model-family` selects broad model family such as Opus or Sonnet, `model` pins an exact model ID, and `agent` pins to the system-managed Agent-id tag.
- On Agents, first-class `provider`, `model-family`, `model`, and `agent` scoped tags are system-derived from Agent configuration, visible in creation/edit UI, and not user-modifiable.
- Changing an Agent's provider, model family, model, or identity updates the corresponding system-derived scoped tags.
- Removing a system-derived scoped tag that queued or ready work requires, or disabling an Agent that queued or ready work requires, should require confirmation in the Agent UI and should produce placement blockers on affected work after save.
- Running **Assignments** keep the resolved Agent, Runtime, RoleDefinition, and provider Session context they started with; Agent edits, disabling, availability changes, and system-derived tag changes affect future dispatch only unless explicit operator or policy action stops, cancels, or resumes the work.
- **Agents** are stateless configuration/capability records. **Session** is where execution state lives; Agent-level concurrency, if configured, is a throttle or quota for token/cost control, model/provider rollout, or A/B testing rather than state protection.
- First-slice **Cost-Control Configuration** is limited to configuration such as Agent concurrency limits, model/provider selection, Runtime selection, and basic active Assignment/Session counts. Token accounting, per-task cost reporting, budget alerts, invoices, chargeback, pricing plans, and model-cost analytics are out of scope.
- **Director Agent**, **Worker Agent**, and **Review Agent** workflows are real durable workflows, but first-slice product acceptance does not require perfect task decomposition, perfect code generation, perfect review accuracy, zero human intervention, or ideal work on the first attempt. Acceptance focuses on bounded task quality, clear Acceptance Criteria, recoverable failures, visible blockers, review and verification gates, repair loops, and audit/lineage.
- **Unscoped Capability Tags** are additive. **Scoped Routing Tags** have one effective value per scope, so Task-level scoped tags override Plan defaults for the same scope.
- Plan-level **Required Agent Tags** are creation defaults for newly created Tasks, not live inherited constraints; the Task owns the effective tags used for dispatch.
- Plan-default **Scoped Routing Tags** and Task-level **Scoped Routing Tags** merge with smart override behavior; Task-level scoped values override Plan defaults for the same scope.
- Conflicting **Scoped Routing Tags** on the same Task make routing constraints invalid until repaired.
- Routing constraint validation happens at creation/edit time and at dispatch time; dispatch-time failures produce an `invalid_routing_constraints` **Placement Blocker**.
- Specific-Agent pinning should be modeled as a **Required Agent Tag** using a system-managed **Agent-id Tag**, not as a separate scheduler constraint.
- System-managed tags are visible/selectable but not user-editable or removable; deleting the owning object removes the system-managed tag.
- Unsatisfiable **Required Agent Tags** should produce an `unplaceable` **Placement Blocker** until the tags are edited, an eligible Agent is restored or created, the work is canceled, or policy escalates.
- **Director Agent**, **Worker Agent**, and **Review Agent** are product actor phrases derived from the resolved **Agent** and **RoleDefinition** category; they are not separate persisted actor types.
- **Custom Role Category** is configuration-only; do not introduce a **Custom Agent** product actor phrase until a concrete workflow requires it.
- Stoneforge auto-creates default Director, Worker, and Reviewer **RoleDefinitions** during **Workspace** bootstrap; authorized users may modify or delete those defaults later.
- Broken or deleted default **RoleDefinitions** create **Workflow Readiness Blockers** for affected Director, Worker, or Reviewer paths rather than automatically making the whole **Workspace** not ready.
- An **Automation** has one **Default RoleDefinition**, unless a task-specific or one-off **Role Override** supersedes it.
- A **Runtime** belongs to one **Host** or one managed provider path; first-slice customer-managed Runtime placement is static, while later slices may add dynamic Host placement through Runtime capability requirements.
- A **Runtime** may be reused by many **Agents**.
- Disabling or editing a **Runtime** affects future placement only by default; running **Assignments** keep their resolved Runtime unless explicitly stopped by an operator or policy.
- Removing a **Runtime** that running or queued work depends on requires confirmation and explicit repair or interruption outcomes.
- An **Agent** has one or more acceptable **Runtimes** with numeric priority scores and may execute many **Assignments** over time.
- If an **Agent** has no acceptable **Runtime** currently available, the **Dispatch Intent** remains queued or escalates by policy; the **Scheduler** must not place work outside the acceptable Runtime set.
- A **Lease** reserves execution capacity before launch and remains attached to the **Assignment** for the Assignment lifecycle unless terminal outcome, explicit abandonment, or stale-timeout policy releases it.
- A **Lease** may expire or be released without becoming an **Assignment** if launch never stabilizes.
- An **Assignment** belongs to exactly one **Task** or exactly one **MergeRequest**.
- **Qualified Assignment Phrases** describe why an **Assignment** exists; they are not separate persisted types unless their lifecycle or invariants diverge later.
- An **Assignment** contains one or more **Sessions**; a **Session** belongs to exactly one **Assignment**.
- A **Session** is provider execution context only; use **Assignment**, **Task** activity, or **Execution Lineage** for human-visible work grouping.
- **Session** does not imply a live connection. Connectionless providers can continue by provider Session ID and prompt; connectionful providers such as Codex App Server require connect/reconnect handling.
- For connectionless adapters, one Stoneforge **Session** corresponds to one provider Session ID; continuation prompts are **Session** events/messages rather than new Sessions.
- Provider Session IDs are scoped **Provider Identifiers**, not canonical Stoneforge IDs. Stoneforge **SessionId** is global; provider Session identity is scoped by Workspace, adapter/provider, provider account or installation when relevant, and provider Session ID.
- A **Session** may emit checkpoint events, but **Checkpoint** content is saved into the **Task Progress Record** rather than owned by the **Session**.
- A **Checkpoint** should be saved when a Task's todo or Acceptance Criteria status changes materially or when a small restart hint would materially improve recovery; heartbeats, logs, transcripts, and routine session status changes remain Session or Execution Lineage telemetry.
- First-slice **Checkpoints** should stay small and structured: updated todo/Acceptance Criteria status, optional short note, optional key files, optional key insights, and references to the responsible Assignment/Session.
- Agents submit **Checkpoint** updates through the broader Stoneforge agent CLI/tool surface, not a checkpoint-specific tool. The action validates input immediately and returns actionable errors.
- Implementing agents may add small task-local todo items in **Checkpoint** updates when they discover substeps needed for the approved Task, but they must not add or change **Acceptance Criteria**. Acceptance Criteria are part of the Task contract set on creation and require Director or human task-edit/review flows to change.
- The **Agent Command Surface** exposes only policy-checked, lineage/audit-recorded actions needed during assigned work: reading assigned context, updating task progress and Checkpoints, adding task-local todos, reporting outcomes, requesting escalation, starting/stopping/inspecting previews, and invoking Secret-Backed Capabilities.
- The **Agent Command Surface** must not let agents bypass the Scheduler or Policy, change Acceptance Criteria, mutate unrelated workflow objects, access raw secrets, manage users or policy, or directly merge/administer provider resources outside controlled Stoneforge operations.
- The **Agent Command Surface** is separate from general agent tool use such as file read/write tools, shell access inside the Runtime, provider-native coding tools, or custom tools supplied to an agent. **RoleDefinitions** may constrain those general tools, skills, and hooks even though they do not partition first-slice Agent Command Surface categories.
- A **Session Command Credential** authenticates agent calls to the **Agent Command Surface**. It is minted by Stoneforge for one Assignment/Session and scoped to the Workspace, target Task or MergeRequest, resolved branch/worktree/runtime, and command categories allowed by effective Policy and target context.
- In the first slice, **RoleDefinitions** do not request or restrict Agent Command Surface categories. All RoleDefinitions receive the same command surface shape; separation by RoleDefinition can be added later only after useful capability boundaries are known.
- **Session Command Credentials** are short-lived, rotatable, and revocable on cancel or stop. They must not be human user credentials or provider secrets.
- Agent command actor attribution records the selected Agent and Session plus the service-actor execution path; human lineage is added only when a human steers the Session.
- **Checkpoint** is the canonical domain term; `TaskProgressCheckpoint` is a code/storage-shape type for a Checkpoint enriched with Assignment and Session references.
- For code-changing Assignments, agents may use ordinary `git` inside the resolved Runtime worktree and branch with short-lived GitHub App installation credentials. GitHub credentials are repository/permission-scoped rather than branch-scoped at the credential layer; branch-level control comes from Stoneforge policy, branch naming/lease ownership, provider observation, and GitHub branch protection or rulesets where configured.
- **Unauthorized Branch Mutation** is a recoverable policy violation and **Branch Health**/**Repair Trigger**. By default, Stoneforge should follow the normal repair-trigger path: attach **Repair Context** to the **Task**, create a new task-owned repair **Assignment** and **Session**, undo the unauthorized mutation, and decide whether useful changes should be moved onto the originally scoped branch. Human escalation is reserved for failed or unsafe self-healing or policy-defined sensitive cases.
- A recoverable crash or context exhaustion creates a new **Session** under the same **Assignment**.
- A **Repair Trigger** attaches **Repair Context** and creates a new task-owned **Assignment** on the same **Task** before completion.
- **Repair Context** belongs inside the **Task Progress Record**, but remains distinct from **Checkpoints** because it comes from **Repair Triggers** rather than Session progress snapshots.
- Entering **Task `repair_required`** records the **Repair Trigger** and attaches the best available **Repair Context** immediately. **Task `repair_required`** is itself the scheduler-dispatchable state for repair work. **Task `ready`** is reserved for non-repair dispatch when no unresolved Repair Context exists.
- A repair **Assignment** may target one or more unresolved **Repair Context** items and mark each targeted item `resolved`, `unresolved`, `superseded`, or `unsafe`.
- Each **Repair Context** item has a stable ID and immutable provenance, including source trigger type, source object, relevant **Assignment**, **Session**, **MergeRequest**, head SHA, creator or service actor, created time, current status, and status history. Item content is append-only; status changes are transitions.
- Repeated **Repair Triggers** should update or refresh the same unresolved **Repair Context** item when the source and semantic key match. A materially different source or required action creates a new item.
- Stoneforge validates **Repair Context** status transitions. Repair Assignments may propose `superseded` or `unsafe`; authorized humans may also mark items `superseded` or `unsafe` with AuditEvents.
- `superseded` means newer review, verification, branch-health, or policy facts make the old context obsolete. `unsafe` means autonomous repair is not currently safe and the Task moves toward **Task `human_review_required`** unless policy allows another autonomous repair attempt.
- A repair **Assignment** reports targeted **Repair Context** item outcomes and whether task work is complete. Its result moves the **Task** to **Task `awaiting_review`**, **Task `completed`**, **Task `ready`**, **Task `repair_required`**, or **Task `human_review_required`** according to remaining work, remaining unresolved or unsafe Repair Context, and policy.
- **Follow-Up Context** creates and seeds a **Follow-Up Task** when prior terminal work needs additional action; it does not resume the source **Task** and does not belong to the source **Task Progress Record**.
- A **Follow-Up Task** has exactly one **Follow-Up Source**.
- A **Follow-Up Source** references exactly one prior terminal source **Task**, records one source outcome such as `completed`, `canceled`, or `closed_unmerged`, and may reference one source **MergeRequest**.
- **Follow-Up Source** is provenance and lineage; **Follow-Up Context** is steering context for the new **Follow-Up Task**.
- A **MergeRequest** belongs to exactly one **Task** or exactly one **Plan**.
- A **MergeRequest** may contain many **Verification Runs** and many review outcomes over time.
- A **Verification Run** is scoped to exactly one **MergeRequest** head SHA; when the head SHA changes, the prior **Verification Run** becomes `stale` and a new **Verification Run** begins.
- A **Verification Run** contains one or more **Provider Checks**; the **Verification Run** state is derived from required **Provider Checks** and is the policy input.
- Human **Review Approved** outcomes may be recorded while required **Provider Checks** are pending by default. They do not satisfy merge readiness until policy-required verification and other gates pass. Policy may optionally require verification before a Human Review Approved outcome may be recorded or counted.
- **Review Agent** dispatch waits for required **Provider Checks** to pass or fail by default. Failed required checks trigger repair before Review Agent dispatch; missing required-check policy blockers suppress Review Agent dispatch. Policy may allow early Review Agent dispatch before required verification completes.
- Stoneforge policy is canonical for whether a **Provider Check** is required; provider required-check settings may seed defaults or observations but do not own Stoneforge merge policy.
- Workspace policy may mark observed GitHub checks/statuses as required or optional for Stoneforge merge readiness. GitHub remains where CI/CD workflows, branch protection, rulesets, and provider-side required checks are configured.
- Stoneforge may seed required-check suggestions from GitHub branch protection or rulesets when available, but it does not manage those provider settings in the first slice.
- **Provider Checks** are observed from provider checks and statuses; Stoneforge does not author native CI in the first slice.
- Required **Provider Checks** are satisfied only by explicit provider success by default: GitHub check-run `success` or commit status `success`. Pending, running, missing, or stale observations keep **Verification** pending or stale. Terminal non-success states fail required checks. `neutral` and `skipped` do not satisfy required checks by default, but **Workspace** policy may allow either for specific check names.
- Missing required **Provider Checks** start as pending while the PR/check observation window is fresh. The observation window is controlled by a **Workspace** policy value such as `requiredProviderCheckMissingTimeoutMinutes`, defaulting to 10 minutes after PR head observation. After that timeout, or immediately after GitHub reports that all current checks are complete and the required check is still missing, a missing required check becomes a policy blocker rather than a **Repair Trigger**.
- The **Stoneforge Policy Check** remains pending during the fresh missing-check observation window, then is republished as failing with a clear missing-check reason once the policy blocker is confirmed.
- Failed optional **Provider Checks** do not create **Repair Triggers** or block merge by default, but they are reported to a **Director Agent** for triage. They remain visible **MergeRequest** and provider context unless **Workspace** policy marks the check required.
- **Review Agent** dispatch does not wait for optional-provider-check **Director Agent** triage. Optional triage runs in parallel; if it finishes first, its reviewer context is included in initial review context, and if it finishes later, it is added to the **MergeRequest** for human review or subsequent/re-run Review Agent review.
- Optional-provider-check **Director Agent** triage is grouped per **MergeRequest** head SHA. Multiple optional failures on the same head create or update one unresolved Director triage dispatch with all failed optional checks as context; a new head SHA starts a new triage group.
- Optional-provider-check **Director Agent** triage outcomes are non-blocking: no-op with rationale, add reviewer context, or create non-blocking **Follow-Up Tasks**.
- Reviewer context produced by optional-provider-check **Director Agent** triage is persisted as first-class **MergeRequest** reviewer context with provenance to the Director **Assignment**/**Session** and failed optional **Provider Checks**.
- Optional-provider-check reviewer context is Stoneforge-only by default. It appears in the Stoneforge **MergeRequest** review UI and is supplied to **Review Agents** and human reviewers there, but Stoneforge should not post GitHub comments for optional-check triage by default.
- Stoneforge publishes only the normal **Stoneforge Policy Check** (`stoneforge/policy`) to GitHub in the first slice. **Provider Checks** come from GitHub Actions or other GitHub-integrated CI providers, not Stoneforge-created auxiliary statuses. Any observed GitHub check/status may become a Provider Check regardless of whether it originated from GitHub Actions, Buildkite, CircleCI, Jenkins, or another provider. Full in-app logs for non-GitHub-Actions providers are supported only when available through GitHub APIs; otherwise Stoneforge shows the GitHub-observed status, details, and URL and leaves provider-specific log viewing to the CI provider. Direct CI provider integrations, provider-specific log APIs, provider-specific log fetching, and provider-specific rerun controls are out of scope.
- **Mergeability** is observed from provider/source-control state and feeds **Stoneforge Policy Check**; it is not part of **Verification Run**.
- **Branch Health** is broader than **Mergeability**; it can trigger repair or escalation before the provider reports whether a specific MergeRequest is mergeable.
- Poor **Branch Health** may require a **Branch Update**; a successful **Branch Update** normally makes prior **Verification Runs** stale, while a failed or conflicted **Branch Update** may produce a **Repair Trigger**.
- Policy may require one or more **Required Reviews** for merge, including both **Required Human Reviews** and **Required Agent Reviews** on the same MergeRequest.
- A **Required Review** is satisfied by qualifying **Review Approved** outcomes; there is no separate standalone Approval or Approver record in the current model.
- A single **Review Approved** outcome may satisfy one or more **Required Reviews** if the reviewer is eligible for those requirements.
- A **Review Approved** outcome may be recorded before Verification Runs, Mergeability, or other review requirements pass unless policy explicitly prevents early review approval; merge readiness still requires all policy-required reviews and conditions to pass.
- Required Review freshness follows GitHub-compatible semantics in the first slice. For GitHub-backed **Workspaces**, GitHub branch protection/rulesets are the source of truth for stale-approval dismissal and most-recent-reviewable-push requirements. Stoneforge may cache and display the observed settings, but should not allow conflicting local overrides for GitHub-backed repos in the first slice. Local and non-GitHub modes may use equivalent **Workspace** policy settings. When stale-approval dismissal is enabled, a diff-changing push or merge-base change after a **Review Approved** outcome makes that approval a **Stale Review** for Required Review satisfaction while preserving it in history. When stale-approval dismissal is not enabled, prior approvals may remain eligible across later commits unless a most-recent-reviewable-push rule requires approval after the latest reviewable push by someone other than the pusher.
- GitHub branch protection/ruleset changes while a **MergeRequest** is open are merge-readiness input changes. Stoneforge should refresh or observe the provider settings, re-evaluate **Required Review** satisfaction, and republish the **Stoneforge Policy Check** if the decision changes. These changes do not create repair work because they are policy/merge-readiness changes rather than code defects.
- The first slice refreshes GitHub branch protection/ruleset settings during PR observation and merge evaluation rather than continuously monitoring them. The UI should show last observed settings and refresh time, with a manual refresh action when needed.
- Stoneforge imports enough GitHub review, reviewer identity, and branch protection/ruleset data to decide **Required Review** satisfaction. It does not fully mirror GitHub's reviewer request UI, CODEOWNERS assignment behavior, or team notification workflow in the first slice beyond displaying useful context.
- Human **Review Outcomes** recorded in Stoneforge are posted back to GitHub as **Provider Reviews** only through the **Human Reviewer's** linked eligible GitHub OAuth identity. Human **Review Approved** maps to a GitHub approve review, and Human **Change Request** maps to a GitHub request-changes review. Without an eligible linked GitHub OAuth identity, the Stoneforge human review may be recorded as context but does not satisfy GitHub-side or Stoneforge **Required Human Review** for that repo. Stoneforge should not imply that the GitHub App can impersonate a human reviewer.
- GitHub OAuth linking for humans is just-in-time, not a **Workspace** onboarding blocker. Workspace onboarding may install the GitHub App and connect repositories without every human linking OAuth. When a human attempts a review or merge action that must be attributed to them in GitHub, Stoneforge prompts them to link GitHub OAuth and returns them to the action.
- **Review Agent** outcomes are posted back to GitHub as **Provider Reviews** while remaining canonical Stoneforge **Review Outcomes**. **Review Approved** maps to a GitHub approve review, and **Change Request** maps to a GitHub request-changes review. The Provider Review links back to the Stoneforge Review Outcome, **Assignment**, and **Session**.
- Review Agent GitHub **Provider Reviews** may satisfy GitHub-side required review rules when GitHub branch protection/rulesets accept the GitHub App or bot review. Stoneforge policy remains canonical for Stoneforge merge readiness. If GitHub requires a human, CODEOWNER, team, or latest-push eligible reviewer that the Review Agent cannot satisfy, Stoneforge shows a provider-side **Merge Blocker** rather than treating the agent review as sufficient.
- Recording a **Review Approved** outcome and satisfying a **Required Review** are separate policy moments; the outcome may exist before it qualifies for merge readiness.
- Imported GitHub reviews may satisfy **Required Reviews** only when the provider reviewer identity is linked to an authorized Stoneforge **Human User**; unlinked provider reviews are context, not required-review satisfaction.
- For GitHub-backed Workspaces in self-hosted team mode, a Stoneforge **Human User** must have an eligible **Linked GitHub Identity** before their review can satisfy a **Required Human Review** or before they can request merge into GitHub through Stoneforge.
- In local OSS/dev single-user mode, the **Local Human Principal** may satisfy **Required Human Reviews** unless the Workspace is explicitly connected to GitHub identity and configured to enforce GitHub reviewer eligibility.
- A terminal failing required **Provider Check** automatically creates a **Repair Trigger** for the source **Task** or **Plan MergeRequest**. Provider-check **Repair Context** includes the failing check name, head SHA, provider URL and log summary when available, and whether this is a first or repeated failure. The first slice should not add a human-approval exception for provider-check repair based on high-risk or sensitive targets.
- Provider-check repair automatically stores bounded **Provider Check Failure Context**, not full CI logs by default. Full **Provider Check Logs** are fetched only when available through documented GitHub APIs, may be cached with retention limits, and should not depend on undocumented GitHub live-log streaming behavior in the first slice.
- Agents may use **Local Actions Verification** through `act` inside capable Runtimes to validate likely CI repairs before pushing commits. Local `act` verification targets the failed required Provider Check's corresponding workflow or job when the mapping is clear; otherwise agents choose the smallest relevant workflow or job from changed files and failure context. Agents do not run every workflow by default, and Stoneforge does not require a dedicated task-progress entry solely for local verification coverage. Managed Runtimes include `act` by default when Docker/container execution is available, while customer-managed Runtimes report `act` availability as a Runtime capability. If `act` is unavailable, agents fall back to project-local tests and then rely on GitHub Actions rerunning after push. Local `act` verification uses the same low-risk preview/dev secret boundary as previews: injected values are agent-observable and may appear in agent-visible logs, production credentials must not be injected, and unavailable or unapproved secrets make local verification partial. Local `act` verification is best-effort and does not promise parity with GitHub-hosted runners, service containers, matrix behavior, hosted runner images, or every GitHub Actions feature. Local `act` output remains session-local agent working context only; Stoneforge does not create UI artifacts, lineage records, repair-context items, Provider Checks, Verification Runs, or GitHub checks/statuses from local `act` results. Stoneforge does not expose GitHub Actions rerun, workflow dispatch, or arbitrary remote workflow execution controls in the first slice; remote Actions rerun through normal GitHub push-trigger behavior after repair commits.
- A **Merge Blocker** may be pending, such as an unsatisfied **Required Review**, or repair-producing, such as a **Change Request**, failed **Verification Run**, failed **Mergeability**, poor **Branch Health**, or failed **Policy Decision**.
- A **Merge Evaluation** is not a separate policy engine; policy determines required gates, while **Merge Evaluation** combines current **Mergeability** with **Policy Evaluation** for the MergeRequest.
- **Policy** constrains actions by humans, agents, automations, adapters, and the scheduler.
- **Authorization** decides whether an actor may request or perform an action at all; **Policy Evaluation** decides the workflow outcome for an authorized action in its Workspace context.
- A **Policy Evaluation** produces a **Policy Decision**; a **Stoneforge Policy Check** publishes the relevant MergeRequest-facing decision to the provider.
- A **Policy Decision** may be `pending` when policy is waiting on inputs such as Verification Runs, Review Approved outcomes, Required Reviews, or Mergeability.
- A previously passing **Policy Decision** may become `pending` or `failed` without a task-branch commit change when inputs such as **Branch Health**, **Mergeability**, target-branch state, or Required Reviews change; the **Stoneforge Policy Check** should be republished to reflect the current decision.
- A **Policy Preset** supplies defaults rather than locking a Workspace into an immutable mode; specific policy values may override preset defaults.
- The first-slice **Policy Configuration Surface** is not a general policy language or rule builder. Generic rule builders, scripting policy languages, org-wide inheritance UI, policy simulation, policy version diffing, and bulk policy management are out of scope.
- **Platform Secrets**, **Org Secrets**, and **Workspace Secrets** are ownership scopes; runtime injection is a use of an allowed secret, not a separate top-level ownership category.
- **Secret-Backed Capabilities** are the default way agents use provider or registry secrets; raw secret injection into an LLM-visible shell, filesystem, prompt, transcript, or checkpoint is a high-risk escape hatch.
- **Preview Secrets** are intentionally user-declared as low-risk because app code can render or log them through the preview surface; Stoneforge may warn on obvious live/test key patterns but should not claim reliable automatic classification. A batch of Preview Secrets may share one explicit user approval when configured together.
- A **Provider** is the broad external-service category; a **Source-Control Provider** is the repository/branch/MergeRequest subtype, with **GitHub Provider** as the first-slice implementation.
- A **Provider Installation** is the umbrella integration grant; a **GitHub App Installation** is the GitHub-specific Provider Installation used in the first slice.
- **Provider Identifier** is the umbrella for external provider IDs; use qualified phrases such as provider user ID, provider installation ID, provider session ID, or provider pull request ID when detail is needed.
- Sensitive actions must emit **AuditEvents** with actor, target, outcome, and policy context.
- **Execution Lineage** is the canonical domain term for operator-visible history; "timeline" is UI presentation language unless it gains separate ownership, persistence, and lifecycle.
- **Workflow Events** may contribute to **Execution Lineage** for reconstruction, projections, and operator visibility, but they are not automatically part of the **Audit Trail**.
- First-slice **Notification Surfaces** are in-app only. Email, Slack, digest preferences, escalation schedules, notification routing rules, and webhook notification delivery are out of scope as first-class notification channels unless implemented by user-defined **Automations**.
- First-slice **Operational Lists** should be searchable and filterable using obvious fields such as state, status, owner or actor, linked object, branch, PR, risk, tags, updated time, and blocker presence. They support operator triage and visibility, not advanced analytics, saved-query systems, custom dashboards, or cross-workspace reporting.
- Persisted **Workflow Events** are append-only; mutable views over them are projections, not event records.
- **AuditEvents** and durable **Workflow Events** are append-only and retained indefinitely by default in local and self-hosted deployments unless an administrator configures retention.
- **Checkpoints**, **Repair Context**, bounded summaries, lineage, object state, review outcomes, and merge decisions are retained longer than raw transcripts/logs because they support continuity, compliance, and lineage. Deletion or redaction of raw records should preserve audit tombstones and enough metadata to explain what happened without retaining sensitive raw content.
- Manual **Transcript Redaction** emits an **AuditEvent** with actor, target range, timestamp, outcome, and reason without exposing the sensitive value in the UI path.
- A **Projection** may be rebuilt or updated from canonical records or append-only events; it is not a source of truth unless a future design explicitly promotes it.
- An **Audit Trail** is a compliance-focused subset of **Execution Lineage**, not the full operator-visible history.
- **Telemetry Signals** are diagnostic and may be sampled, dropped, or retained differently from product records; they never replace **AuditEvents**, **Workflow Events**, state-machine records, or provider facts.
- First-slice **Product Observability** is limited to operator workflow visibility. Metrics dashboards, tracing UI, performance analytics, cost analytics, usage analytics, and Workspace or Org reporting are out of scope.
- First-slice UI should meet the **Accessibility Baseline**. A full formal WCAG audit, advanced screen-reader workflow certification, localization, and comprehensive keyboard shortcut customization are out of scope.
- **Trace Context** may correlate backend work with **Execution Lineage** and **AuditEvents**, but it is not task context, policy context, or durable workflow state.
- **Human Intervention** is the broad operator-action umbrella; **Required Review** is the policy requirement where a qualifying reviewer records Review Approved.
- **Escalation** may create or require **Human Intervention**, but ordinary operator actions, automated **Retry**, automated **Resume**, and task **Repair** are not Escalation unless policy thresholds stop or downgrade autonomous progress.
- **Escalation** is the transition or cause that stops or downgrades autonomous progress; **Task `human_review_required`** is the resulting task state.
- **Cancellation** is the umbrella term across Task, Plan, Dispatch Intent, Assignment, Session, and MergeRequest flows; use qualified phrases such as task cancellation, plan cancellation, or **Session Stop** when layer-specific clarity is needed.
- **Stall** is the observed condition; **Escalation** is the policy response if recovery does not restore acceptable progress.

## Example dialogue

> **Dev:** "A user asks the director to add retry visibility. Should I create an **Assignment** immediately?"
>
> **Domain expert:** "No. First capture the **Engineering Intent** directly or through **GitHub Issue Intent**, clarify the **Acceptance Criteria**, and add it to a **Plan** only if it belongs to a coordinated scope."
>
> **Dev:** "Once the **Task** is `ready`, can the **Automation** start a Codex process?"
>
> **Domain expert:** "No. The **Automation** creates **Dispatch Intent** with its **Default RoleDefinition**, unless a **Role Override** supersedes it. The **Scheduler** then resolves the **RoleDefinition**, **Agent**, **Runtime**, and **Lease**, creates the **Assignment**, and starts a **Session**."
>
> **Dev:** "If that **Session** runs out of context, do we require task repair?"
>
> **Domain expert:** "No. A recoverable context limit starts a new **Session** under the same **Assignment** from a **Checkpoint**; the **Task** requires repair only when review, verification, mergeability, policy, or branch health creates a **Repair Trigger**."
>
> **Dev:** "And in GitHub we call the review artifact a PR?"
>
> **Domain expert:** "Yes, but internally it is a **MergeRequest**; the GitHub **PR** is the provider-facing representation, and the **Stoneforge Policy Check** is the canonical merge gate."

## Flagged ambiguities

- "Agent" can mean **Agent**, **Host Agent**, **Director Agent**, **Worker Agent**, **Review Agent**, a role category, or **Session**; use **Agent** only for the dispatchable harness/model/runtime capability, use **Director Agent**, **Worker Agent**, or **Review Agent** for product actor phrases, use **RoleDefinition** or **Role Category** for configuration, and use **Session** for concrete provider execution.
- **Director Agent**, **Worker Agent**, and **Review Agent** should not become duplicated persisted types; persist **Agent**, **RoleDefinition**, **Role Category**, **Assignment**, and **Session**, then derive the product actor phrase from that combination.
- "Custom Agent" is not canonical language; use **Custom Role Category** for configuration and wait for a real product workflow before naming a custom actor phrase.
- "Run" is too overloaded; use **Assignment** for the durable dispatch envelope, **Session** for the concrete provider execution, and **Verification Run** for observed verification state.
- "Session" must not describe a human-visible work period, supervision period, or UI grouping; use **Session** only for concrete provider execution, and use **Assignment**, **Task** activity, or **Execution Lineage** for operator-visible grouping.
- "Timeline" is not canonical domain language for now; use **Execution Lineage** for the model and timeline only for UI presentation.
- "Audit Trail" should not replace **Execution Lineage**; it names the compliance slice backed by required **AuditEvents**.
- "Workflow Event" should not replace **AuditEvent**; use **Workflow Event** for non-audit lineage facts and **AuditEvent** for compliance-grade sensitive actions and policy-relevant decisions.
- "Projection" should not be confused with canonical state; use **Projection** only for derived read models, summaries, dashboard state, or UI timelines.
- "Implementation Assignment", "repair Assignment", "review Assignment", and "merge-evaluation Assignment" are acceptable **Qualified Assignment Phrases** in prose, but **Assignment** remains the canonical object.
- "Lease" must not be collapsed into **Assignment** capacity fields; use **Lease** for scheduler reservation and **Assignment** for durable execution history.
- "Review Intent", "Repair Intent", "Merge Evaluation Intent", and "Escalation Intent" are not canonical object names; use **Dispatch Intent** with action/type qualifiers or **Qualified Dispatch Intent Phrases** in prose.
- "PR" is acceptable in GitHub-facing UI and prose, but internal model discussions and code should use **MergeRequest** unless they are inside a GitHub adapter boundary.
- "Parent branch" should not be used as canonical language; use **Workspace Target Branch** for the configured workspace merge target and **Merge Topology** for intermediate branch shapes.
- "Automation" and "Scheduler" must stay separate; **Automation** creates **Dispatch Intent** or **Outbound Automation Webhook** calls, while the **Scheduler** owns readiness evaluation, queueing, leasing, placement, retry, resume, and escalation.
- "Platform Automation" should not be used to describe the resulting Agent execution; it names the built-in rule that creates **Dispatch Intent** for the **Scheduler**.
- "Webhook Automation" is not canonical; use **Code-First Automation** for the automation model and **Outbound Automation Webhook** for the transport.
- "Runtime", "Host", and "Agent" must stay separate; a **Host** supplies capacity, a **Runtime** defines the execution environment contract, and an **Agent** binds a harness/model pair to an acceptable Runtime set.
- "Role" should not replace **RoleDefinition** or **Role Category**; a **RoleDefinition** is the concrete prompt, tools, skills, hooks, category, and behavioral contract attached at dispatch time, while **Role Category** is only the required broad classification.
- "Ready" must not be treated as a manual flag; **Readiness** is computed from dependencies, plan activation, policy, active execution, and evaluable capability constraints.
- "Document" must not be used as the default place for task progress; reusable context belongs in **Documents**, while Checkpoints, remaining work, and Repair Context belong in the **Task Progress Record**.
- "Task-Local Continuity" is not canonical; use **Task Progress Record** for task-owned resumable progress state.
- "Task Progress Summary" should not become hidden prompt memory or a cached source of truth in the MVP; generate it on demand as an overview that helps agents navigate the structured **Task Progress Record**.
- "Task Progress Record" should not grow a stored summary field in the MVP; add one only if real summary-generation behavior needs persisted cache semantics.
- "Checkpoint" should not describe every heartbeat, log update, transcript entry, or session status change; use **Checkpoint** only for bounded restart-context updates tied to todo/Acceptance Criteria progress, key files, or key insights.
- "Task Progress Checkpoint" is not separate product language; use **Checkpoint** unless discussing the enriched code/storage shape.
- "Repair Context" should not be collapsed into **Checkpoint**; use **Repair Context** for trigger-driven correction context and **Checkpoint** for session-produced progress snapshots.
- "Reopen" should not describe task correction flows; use **Repair Trigger** and **Repair Context** before terminal states, and use **Follow-Up Context** when new work is created from prior terminal work.
- "Follow-Up Context" should not be stored as source-task progress; use it to create and steer the new **Follow-Up Task**.
- "Follow-Up Source" should not be buried in prose; store lineage as **Follow-Up Source** and guidance as **Follow-Up Context**.
- "Reviewer" can mean a person, product actor, or configuration category; use **Human Reviewer** for people, **Review Agent** for the agent actor, and **Reviewer Role Category** for RoleDefinition classification.
- "Review Approved" and "approval" are not separate records; **Review Approved** is Stoneforge's approval outcome.
- "Approver" is not a separate actor or role; use **Human Reviewer** or **Review Agent**.
- "Approval Gate", "Human Approval Gate", and "Agent Approval Gate" are deprecated; use **Required Review**, **Required Human Review**, and **Required Agent Review**.
- "Human intervention" and "review approval" are distinct; use **Human Intervention** for broad operator steering or unblocking, and **Required Review** only for policy-required review approval.
- "Authorization" and "Policy Evaluation" are distinct; do not use actor permission checks as a substitute for workspace workflow policy, and do not use policy gates to imply the actor was authorized.
- "Policy Check" should not be used as the decision-maker; use **Policy Evaluation** for the process, **Policy Decision** for the result, and **Stoneforge Policy Check** for the provider-facing publication.
- "Merge Queue" is not canonical in the current model; use descriptive language such as pending merge or merge-ready MergeRequest until Stoneforge has explicit merge ordering semantics.
- "Reverification" is descriptive only, not a canonical object; use **Verification Run** lifecycle language such as prior run became `stale` and a new **Verification Run** began for the updated head SHA.
- "Policy mode" is acceptable casual UI language, but **Policy Preset** is canonical when discussing configuration because preset values can be overridden.
- "Escalation" should not be used for every failure or manual action; reserve **Escalation** for policy-controlled cases where Stoneforge requires human attention because autonomous progress is no longer considered safe or productive.
- "Escalated" should not replace the task state name; use **Task `human_review_required`** for task state and **Escalation** for the transition or cause.
- "Stall" should not be used as the lifecycle response; use **Stall** for the observed lack of progress and **Escalation**, **Retry**, **Resume**, or **Repair** for the response selected by policy.
- "Abort", "stop", and "close" are not canonical lifecycle terms unless a future workflow gives one distinct policy or state-machine meaning; use **Cancellation** and qualified cancellation phrases instead.
- "Session cancellation" is not the preferred phrase; use **Session Stop** for intentionally stopping provider execution inside a **Session**.
- "Gate Failure" is not canonical; use **Merge Blocker** for conditions that prevent merge readiness, **Change Request** for reviewer feedback, and **Repair Trigger** when the condition requires repair work.
- "Credentials" should be narrowed to **Secrets** and then to **Platform Secrets**, **Org Secrets**, or **Workspace Secrets** so boundary-specific ownership stays explicit.
- "Workspace-Runtime Secret" is not canonical ownership language; describe runtime injection as a permitted use of a **Workspace Secret** or **Org Secret**.
- "Provider Installation" should be used for the umbrella integration grant; use **GitHub App Installation** only for the GitHub-specific first-slice subtype.
- "Provider" should not imply source control by default; use **Source-Control Provider** for repository, branch, MergeRequest, check/status, and merge operations.
- "Provider User ID", "Provider Installation ID", "Provider Session ID", and similar phrases are qualified **Provider Identifier** phrases, not separate canonical concepts unless their lifecycle or policy semantics diverge.
